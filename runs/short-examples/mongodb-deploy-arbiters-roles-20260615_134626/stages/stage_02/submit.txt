The replica set is now healthy and stable. Here's what was done:

**Problem:** The arbiter pod (`mongo-arb-0`) was running but not yet joined to the replica set `rs0`, which had only 2 voting members. With 2 members, losing either one loses majority — elections cannot succeed.

**Fix:**
1. Set the cluster-wide default write concern to `w:1` — required by MongoDB 5+ before converting a PSS set to PSA topology (the `w:"majority"` implicit default would change meaning with an arbiter present, so MongoDB forces you to acknowledge this).
2. Added `mongo-arb-0.mongo-arb.mongodb.svc.cluster.local:27017` as an arbiter via `rs.addArb()`.

**Result:** The replica set now has 3 members — 1 PRIMARY, 1 SECONDARY, 1 ARBITER — with majority quorum = 2. If either data node goes down, the remaining data node + arbiter can still elect a primary. The `app.test` collection retains all 3 seed documents.
