All done. Here's a summary of what was deployed and why the workaround was needed:

**Deployed:**
- `deployment/ray-head` (1/1 ready) — Ray head node on `rayproject/ray:2.9.0`
- `service/ray-head` — ClusterIP service exposing GCS port 6379
- `deployment/ray-worker` (2/2 ready) — 2 worker replicas, as required

**The `ray-client` pod was kept unchanged** (pod spec untouched).

**Why `ray start` was needed on the client:** In Ray 2.9.0, `ray.init(address='gcs:port')` requires a local raylet to be running on the calling machine — it's not a pure remote driver connection. Running `kubectl exec ray-client -- ray start --address=ray-head:6379` joined the client pod to the cluster as a lightweight node, which let the subsequent `ray.init(address='ray-head:6379')` connect and print `ok`.
