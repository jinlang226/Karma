## Diagnosis & Fix Summary

### Root Cause: Data Skew + Insufficient Executor Memory

**Data skew (confirmed via `/data/skew_info.txt`):**
- Total records: 100,000
- `user_id 999` holds: **50,000 records (50% of all data)**
- Skew ratio: **500x** (far beyond the 10x threshold)

**Why it caused OOM:** Stage 3 does a `GROUP BY user_id`. With 50% of rows mapping to a single key, one executor receives half the dataset. At 256MB executor memory, that partition exceeds available heap and throws `OutOfMemoryError`.

### Fix Applied (Option C — AQE + increased executor memory)

The `etl-job.yaml` was updated with:
```yaml
--conf spark.executor.memory=1g                        # was 256m
--conf spark.sql.adaptive.enabled=true                 # AQE on
--conf spark.sql.adaptive.skewJoin.enabled=true        # splits hot partitions
```

AQE's skew-join handling automatically detects oversized partitions at runtime and splits them across multiple tasks, distributing `user_id=999`'s 50,000 records rather than dumping them on one executor.
