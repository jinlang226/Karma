The `config-audit` ConfigMap is created and verified. Here's a summary of the audit findings:

---

**6 PASSED controls:**
- Dedicated `spark-pi` ServiceAccount — no workload uses `default`
- Namespace-scoped RBAC only (no ClusterRole/ClusterRoleBinding)
- Resource requests + limits set on every container (200m/512Mi → 500m/1Gi)
- Image pinned to `apache/spark:3.5.3` (not `latest`)
- Job `restartPolicy: Never` + `backoffLimit: 0`
- Job completed successfully

**7 FAILED controls:**
1. **No securityContext** anywhere — containers likely run as root; no `runAsNonRoot`, no `allowPrivilegeEscalation: false`, no capability drops
2. **No NetworkPolicy** — unrestricted lateral movement between all pods cluster-wide
3. **Image not digest-pinned** — `apache/spark:3.5.3` is a mutable tag; susceptible to tag-reassignment supply-chain attacks
4. **RBAC role overly permissive** — grants `delete` on pods + write on configmaps/PVCs; the `--master local[2]` SparkPi job needs none of that
5. **`imagePullPolicy: IfNotPresent`** — stale/compromised images won't be re-pulled on cached nodes
6. **No namespace ResourceQuota** — only per-container limits; no cap on total namespace resource consumption
7. **`default` SA not disabled** — `automountServiceAccountToken` not set to `false`, leaving a token-mount footgun for inadvertently created pods

No changes were made to the cluster — this was a strictly read-only audit.
