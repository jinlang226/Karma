Done. Elections are now stable.

**What was wrong:** The replica set `rs0` had only the two data-bearing members (`mongo-rs-0`, `mongo-rs-1`) configured — 2 voting members, an even number. The arbiter pod (`mongo-arb-0`) was running but had never been added to the replica config, so elections could deadlock on a tie.

**Fix:** Added the existing arbiter pod as a voting **arbiter** member:
- `rs.addArb()` initially failed because adding an arbiter to a 2-member set changes the implicit default write concern, so MongoDB required an explicit cluster-wide write concern first. Set `setDefaultRWConcern` → `{w:"majority"}`, then added the arbiter.

**Result:**
- 3 voting members, **3 total votes** (odd → stable elections), but still only **2 data-bearing members** — the arbiter holds no data, so no added storage cost.
- `mongo-rs-0` PRIMARY, `mongo-rs-1` SECONDARY, `mongo-arb-0` ARBITER.
- App database intact: `app.test` still has all **3 documents**.
