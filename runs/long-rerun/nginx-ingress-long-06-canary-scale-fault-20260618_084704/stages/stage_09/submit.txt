Rate limiting for rate.example.com/api is configured and verified.

Configuration:
- Ingress `rate-api` (demo ns, path /api) carries:
    nginx.ingress.kubernetes.io/limit-rps: "1"
    nginx.ingress.kubernetes.io/limit-burst-multiplier: "1"
  This rejects excess requests when probed at 2 QPS (every other request
  is over budget), producing ~4-5/10 rejections.
- Ingress `rate-health` (path /health) has NO limit annotations, so /health
  is never throttled.
- Controller ConfigMap ingress-nginx/ingress-nginx-controller sets
    limit-req-status-code: 429
  so rate-limited responses return HTTP 429 (not the nginx default 503).
- The application Deployment was not modified.

Verification (kubectl exec curl-test, 2 QPS):
- /api: repeated runs returned 4-5 of 10 responses as 429 (>= 4 required).
- /health: 10/10 returned 200.
