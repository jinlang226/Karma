## Results

### Bug Fixes Applied

1. **AQE `skewedPartitionThresholdInBytes` misconfiguration** тАФ The Python script in `skew-job-aqe.yaml` set this to `"1b"` (1 byte), causing Spark to treat *every* partition as skewed and split them all тАФ massive overhead instead of improvement. Fixed to `"256mb"` (the Spark 3.x default).

2. **Resource contention from concurrent execution** тАФ The broadcast and AQE jobs ran simultaneously when first applied, competing for limited CPU (500m limit each). Re-ran them sequentially for accurate comparison.

### How Each Strategy Works

- **Baseline**: `repartition(20, "product_id")` creates severely skewed partitions (max 11,579 rows vs min 263). The sort-merge join stalls on the hot-product partitions.
- **Broadcast join**: Broadcasts the small `products` table (100 rows) to all executors тАФ eliminates the shuffle entirely, so skew in `orders` doesn't matter.
- **AQE skew join**: Spark detects skewed partitions at runtime and auto-splits them, then reads matching build-side data multiple times to compensate. With `256mb` threshold (correctly sized for this dataset), no unnecessary splitting occurs and the skewed partitions are handled adaptively.
