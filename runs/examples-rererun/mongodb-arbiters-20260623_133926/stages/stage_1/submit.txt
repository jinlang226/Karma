Done. Here's what was done and why:

**Problem:** The arbiter pod (`mongo-arb-0`) was running but not registered in the replica set. With only 2 voting members (PRIMARY + SECONDARY), majority = 2, so losing either member makes elections impossible.

**Fix:**
1. Set `defaultWriteConcern: {w: 1}` cluster-wide — required by MongoDB when reconfiguring a set in a way that changes the implicit default write concern (adding an arbiter reduces data-bearing members' majority).
2. Called `rs.addArb('mongo-arb-0.mongo-arb.mongodb.svc.cluster.local:27017')` to register the already-running arbiter as a voting, non-data-bearing member.

**Result:**
- `votingMembersCount: 3`, `majorityVoteCount: 2` — now any single member can fail and elections still proceed
- No additional storage: the arbiter holds no data
- `app.test` still has 3 documents — data untouched
