All verification checks pass:

- **All 3 pods**: `1/1 Running` (no restarts on the fixed pod)
- **Node status**: all 3 nodes `is_available=true`, `is_live=true`
- **SQL**: `SELECT 1` succeeds

**Root cause**: The StatefulSet command contained a conditional that bound `crdb-cluster-1`'s HTTP server to `127.0.0.1:8080` (loopback only) instead of `0.0.0.0:8080`. Kubernetes health probes connect to the pod's IP address, not localhost, so both liveness and readiness probes got "connection refused" — liveness killed the pod in a loop, readiness kept it out of the Service. The fix was patching the StatefulSet to always use `--http-addr=0.0.0.0:8080`, which triggered a rolling update and brought `crdb-cluster-1` back healthy.
