**DIAGNOSIS: Root Cause**

The data skew report confirms:
- `total_records=100,000`, `distinct_user_ids=1,000`
- `hot_user_id=999` holds `50,000` records — **50% of all data**
- `skew_ratio=500x` (far above the 10x threshold)

When Stage 3 runs `GROUP BY user_id`, Spark shuffles all data for `user_id=999` into a single partition. With only 256MB of executor memory, that one massive partition triggers `OutOfMemoryError`.

**FIX: What was applied (etl-job.yaml:30-37)**

The YAML already contains the fix — a combination of Option B + Option C:

```yaml
--conf spark.executor.memory=1g                      # ↑ from 256m (4x increase)
--conf spark.sql.adaptive.enabled=true               # AQE on
--conf spark.sql.adaptive.skewJoin.enabled=true      # AQE splits hot partitions
```

AQE's skew join detection is the right long-term fix: it automatically splits the oversized `user_id=999` partition into sub-partitions at runtime, so no single executor is overwhelmed regardless of skew severity.
