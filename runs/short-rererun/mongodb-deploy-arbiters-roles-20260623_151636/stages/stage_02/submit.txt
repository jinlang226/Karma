Done. Here's what was done and why:

**Root cause:** The replica set was initialized with only the 2 data-bearing members (`mongo-rs-0`, `mongo-rs-1`). A 2-node set requires both members for a majority (1/2 < majority), so any single node failure makes elections impossible.

**Fix:**
1. Set an explicit cluster-wide write concern (`{w: 1}`) — required by MongoDB 5.0+ before reconfiguring to a PSA (Primary-Secondary-Arbiter) topology.
2. Added `mongo-arb-0` as an arbiter via `rs.addArb()`.

**Result:**
- 3 voting members total: 2 data-bearing (votes=1 each) + 1 arbiter (votes=1, `arbiterOnly=true`)
- Majority quorum = 2, so elections succeed even when one data node is down
- Arbiter stores no data — zero storage cost increase
- `app.test` still has all 3 documents — data intact
