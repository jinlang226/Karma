## Summary

A 3-member authenticated MongoDB replica set is now running in the `mongodb` namespace.

**What was created:**
- **StatefulSet `mongodb-replica`** — 3 pods `mongodb-replica-0..2` (all `Running`, `3/3` ready), running `mongod --replSet mongodb-replica --keyFile ... --bind_ip_all`.
- **Headless Service `mongodb-replica-svc`** — `clusterIP: None`, governing the StatefulSet so members resolve at `mongodb-replica-<n>.mongodb-replica-svc.mongodb.svc.cluster.local`.
- **Replica set `mongodb-replica`** — initiated with that exact `_id` (not the `rs0` default), members addressed by their stable DNS names. Status: **1 PRIMARY + 2 SECONDARY**.
- **Keyfile auth** — a random 1024-byte keyfile stored in Secret `mongodb-keyfile`; an initContainer copies it into an emptyDir with `chmod 400` / `chown 999:999` so mongod accepts it for internal replica-set authentication.
- **Users** (created via the localhost exception, then admin-authenticated):
  - `admin-user` / `admin123password` (root on `admin`)
  - `app-user` / `app123password` (readWrite on `appdb`)
- **Secrets** holding the passwords under key `password`: `admin-user-password`, `app-user-password`.

**Verification:**
- Replica set healthy (PRIMARY + 2 SECONDARY, set name `mongodb-replica`).
- As `app-user`, inserted a document into `appdb.items` and read it back (count = 1).
- Confirmed the write replicated — reading from secondary `mongodb-replica-1` also returns count = 1.
