Done. Elections are now stable.

**What was wrong:** The `rs0` replica set had only two voting, data-bearing members (`mongo-rs-0`, `mongo-rs-1`) — an even vote count, which causes election instability (no majority can be formed if one member is unreachable). A `mongo-arb-0` arbiter pod was already running with `--replSet rs0` but had never been added to the config (`NotYetInitialized`).

**Fix applied:**
1. Set a cluster-wide default write concern (`w:"majority"`) — required because MongoDB 6.0 blocks adding an arbiter (PSS→PSA) that would change the implicit default write concern.
2. `rs.addArb("mongo-arb-0.mongo-arb.mongodb.svc.cluster.local:27017")` — added the existing arbiter pod as a voting, `arbiterOnly` member.

**Result:**
- Voting members now 3 (odd) → stable elections: `mongo-rs-0` PRIMARY, `mongo-rs-1` SECONDARY, `mongo-arb-0` ARBITER, all `health:1`.
- The arbiter holds **no data** (`arbiterOnly:true`), so no storage cost was added and no data-bearing member was introduced.
- App database intact: `app.test` still has 3 documents (unchanged before/after).
