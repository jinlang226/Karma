The 3-node CockroachDB cluster is fully deployed and healthy. Here's a summary of what was done and one non-obvious fix applied:

**Resources created:**
- `crdb-cluster` — headless Service (ClusterIP: None), ports 26257/8080, `publishNotReadyAddresses: true`
- `crdb-cluster-public` — regular ClusterIP Service, ports 26257/8080
- `crdb-cluster` — StatefulSet (3/3 Ready), `podManagementPolicy: Parallel`, image `cockroachdb/cockroach:v23.2.0`, 10Gi PVC per pod
- `crdb-cluster-pdb` — PodDisruptionBudget with `minAvailable: 2` (preserves quorum)

**Node status:** all 3 nodes live and available, each advertising their full headless DNS name.

**Two non-obvious bootstrapping issues encountered and fixed:**
1. `podManagementPolicy: Parallel` — required because the default `OrderedReady` policy waits for pod N to be Ready before starting pod N+1, but CockroachDB pods can't become Ready until the cluster is initialized, which requires all pods to be running.
2. `publishNotReadyAddresses: true` on the headless service — required because Kubernetes only creates per-pod DNS A records for Ready pods by default. Pod-0 couldn't resolve pod-1/2's hostnames (needed for gRPC gossip back-connections), breaking cluster join. Setting this flag makes all pod DNS records available regardless of readiness state.
