### Performance Comparison

### What the skew looks like
The baseline showed a **44x skew ratio**: the largest partition held 11,579 rows while the smallest held only 263. Hot products (ids 1–5) each drew ~25% of all even-numbered orders, creating massive straggler partitions.

### How each optimization helps
- **Broadcast join**: eliminates the shuffle that creates skewed partitions — the small `products` table (100 rows) is sent to every executor, so each task locally joins without any data movement.
- **AQE skew join**: Spark detects the skewed partitions after the shuffle stage and automatically splits large partitions into sub-tasks, spreading the straggler work across multiple executors.

Both strategies cut runtime roughly in half. For this workload (small products table), broadcast join is the simplest fix; AQE is more general and works automatically without knowing table sizes in advance.
