Done. Here's what was done and why:

**Root cause:** The replica set was initiated with only 2 voting members (`mongo-rs-0`, `mongo-rs-1`). With 2 votes, reaching majority requires both nodes — losing either one makes elections impossible. The arbiter pod was running but not registered in the replica set config.

**Fix applied:**
1. `setDefaultRWConcern` — MongoDB 5.0+ requires an explicit cluster-wide write concern before any reconfig that changes the implicit majority (adding a non-data-bearing arbiter shifts the data-bearing majority from 2→1).
2. `rs.addArb("mongo-arb-0.mongo-arb.mongodb.svc.cluster.local:27017")` — added the arbiter as `_id: 2`, `arbiterOnly: true`.

**Result:**
- 3 voting members: PRIMARY (`mongo-rs-0`), SECONDARY (`mongo-rs-1`), ARBITER (`mongo-arb-0`)
- Majority is now 2 of 3 — the set can survive losing one member and still elect a primary
- No storage added (arbiters hold no data)
- `app.test` still has 3 documents — data intact
