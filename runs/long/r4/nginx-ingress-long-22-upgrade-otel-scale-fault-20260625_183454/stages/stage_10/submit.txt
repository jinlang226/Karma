Both routes work correctly. Here's what happened and what fixed it:

**Root cause:** Both `canary-stable` and `canary-canary` were using `ingressClassName: ingress-1` (correct for this cluster — the controller at `ingress-nginx-controller.ingress-nginx.svc` runs with `--ingress-class=ingress-1`). However, the canary ingress wasn't actively registered in the controller's config, likely due to a stale/partial state when the scenario was set up.

**Fix:** Patching both ingresses to `ingressClassName: nginx` (wrong class) and then reverting back to `ingressClassName: ingress-1` forced the nginx controller to evict and re-add both ingresses, triggering a full config rebuild. With both ingresses freshly registered:

- `canary-stable` → primary ingress for `canary.example.com` → routes to `stable-echo`
- `canary-canary` → canary ingress with `nginx.ingress.kubernetes.io/canary-by-header: X-Canary` / `canary-by-header-value: always` → routes `X-Canary: always` requests to `canary-echo`

**Results:**
- Without header: `stable` ✓
- With `X-Canary: always`: `canary` ✓
