## Summary

**Root cause:** After the "permissions change," `reporting-user` had been granted a custom role `rawRead` that allowed `find` on **`appdb.raw`** (the ingestion data that must stay private) and had **no** access to `appdb.reports` — so dashboards (which read `appdb.reports`) failed, while the sensitive `raw` collection was improperly exposed.

**Fix (least privilege):**
1. Created a minimal custom role `reportsRead` on `appdb` granting only `find` on `appdb.reports`.
2. Granted `reportsRead` to `reporting-user` and revoked `rawRead`.
3. Resynced `reporting-user`'s password to match the `reporting-user-password` Secret (its DB password was out of sync, blocking authentication entirely).

**Verification** (authenticating as `reporting-user` with the Secret's password, `authSource=admin`):
- `appdb.reports` → readable (4 docs) ✅
- `appdb.raw` → **Unauthorized** ✅

No operators/CRDs used; admin password and replica set were left untouched.
