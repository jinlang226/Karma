## Summary

Deployed into the `cockroachdb` namespace using only native Kubernetes resources:

| Resource | Name | Details |
|----------|------|---------|
| Headless Service | `crdb-cluster` | `clusterIP: None`, exposes grpc/26257 + http/8080, `publishNotReadyAddresses: true` for peer discovery |
| Public Service | `crdb-cluster-public` | ClusterIP, exposes grpc/26257 + http/8080 |
| StatefulSet | `crdb-cluster` | 3 replicas, image `cockroachdb/cockroach:v23.2.0` |
| PDB | `crdb-cluster-pdb` | `maxUnavailable: 1` — preserves quorum (2 of 3) on voluntary disruptions |

Key configuration choices:
- **Per-pod DNS advertise**: `--advertise-host=$(hostname).crdb-cluster.cockroachdb.svc.cluster.local`
- **Join all 3 nodes**: `--join=crdb-cluster-0,1,2.crdb-cluster...:26257`
- **Insecure mode**: `--insecure` (no TLS, as required)
- **Storage**: volumeClaimTemplate `datadir` with 10Gi (all 3 PVCs Bound)
- **RBAC**: reused existing `crdb-cluster-sa`; no new RBAC, no CRDs, no operator
- **PodDisruptionBudget** added to satisfy the quorum-preservation constraint during voluntary disruptions (drains/evictions)

After applying, I ran the one-time `cockroach init --insecure` (required to bootstrap a fresh `--join` cluster). All three pods are now `1/1 Running`, the StatefulSet reports `3/3`, and `node status` confirms all 3 nodes are live and available.
