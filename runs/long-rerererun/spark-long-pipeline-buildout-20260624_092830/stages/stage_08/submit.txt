## Step 3: Broadcast Join Optimization Applied

kubectl apply -f cases/spark/spark_data_skew/resource/skew-job-broadcast.yaml

strategy=broadcast  elapsed=14.61s
optimization=broadcast_join  benefit=no_shuffle_no_skew

Improvement: 24.59s → 14.61s (40.6% faster)
Mechanism: products table broadcast to all executors; no shuffle needed,
so the skew in product_id distribution never causes partition imbalance.

## Step 4: AQE (Adaptive Query Execution) Applied

kubectl apply -f cases/spark/spark_data_skew/resource/skew-job-aqe.yaml

strategy=aqe  elapsed=13.5s
optimization=aqe_skew_join  benefit=auto_split_hot_partitions

Improvement: 24.59s → 13.5s (45.1% faster)
Mechanism: Spark AQE detects skewed partitions at runtime and automatically
splits them, distributing hot-product data across multiple tasks.
