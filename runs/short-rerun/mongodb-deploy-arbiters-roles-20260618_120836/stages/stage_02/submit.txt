Done. Elections are now stable.

**What was wrong:** The replica set `rs0` had only two data-bearing members (both voting). An even number of voters can't reliably break ties, so elections were unstable. A `mongo-arb-0` arbiter pod was running but was **not** a member of the replica set config.

**Fix applied:**
1. Added `mongo-arb-0.mongo-arb.mongodb.svc.cluster.local:27017` to the replica set as an **arbiter** (`rs.addArb`). An arbiter votes but stores no data → odd vote count (3), no added storage cost, and no new data-bearing member.
2. The first attempt was rejected because moving from a 2-member set to a Primary-Secondary-Arbiter topology changes the implicit default write concern. I set an explicit cluster default (`setDefaultRWConcern` → `w:"majority"`) and the reconfig succeeded.

**Verified end state:**
- Members: `mongo-rs-0` PRIMARY, `mongo-rs-1` SECONDARY, `mongo-arb-0` ARBITER — all `health=1`, 3 votes total.
- App data intact: `app.test` still has 3 documents.
