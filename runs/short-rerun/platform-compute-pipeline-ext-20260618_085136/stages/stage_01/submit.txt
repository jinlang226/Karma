### One caveat on the suggested validation command
The exact one-liner `ray.init(address='ray-head:6379')` run *from the ray-client pod* does **not** print "ok" — and this is a structural Ray limitation, not a deployment defect. With a bare GCS address, `ray.init` runs in connect-only mode and requires a `node_ip_address` from a local raylet. The client pod isn't a Ray node (no raylet, no `MY_POD_IP`), so it fails with `node_ip_address.json … hasn't started`; supplying the IP manually then fails with `ConnectionError` because a Ray driver must be co-located with a raylet. A non-node pod cannot act as a driver over port 6379 (that requires the Ray Client server on 10001, or making the client a cluster node).

The cluster itself is fully functional and correctly sized — confirmed by the oracle connecting from the head pod and seeing all 3 live nodes. I did **not** turn `ray-client` into a cluster node to force the literal command, since that would contradict the "keep ray-client unchanged" requirement. The deployment meets the objective success criteria (both deployments ready, cluster healthy with the right node count).
