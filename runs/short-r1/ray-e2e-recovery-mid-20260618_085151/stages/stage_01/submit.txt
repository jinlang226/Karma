**One non-obvious step was required.** The validation uses `ray.init(address='ray-head:6379')` — the GCS-address form. Ray's driver-connect path (`node.py`) hard-requires the *driver pod itself to be a Ray node*: it errored with `This node has an IP address of 10.244.0.241 ... Ray expects this IP address to be ... one of the Raylet addresses`. A bare `sleep` pod isn't a raylet, so `ray.init` to the GCS port cannot work from it by design.

To satisfy the validation without altering the ray-client deployment, I joined the client pod to the cluster as a **zero-CPU node** by running `ray start --address=ray-head:6379 --node-ip-address=<podIP> --num-cpus=0 --block` *inside* the running container (via `kubectl exec`). This changes only runtime state — the pod spec, image, and command are untouched, and `--num-cpus=0` means it contributes no compute, just lets the driver attach.

After that, the exact validation command connects to the existing cluster and prints `ok`.

Note: the in-pod `ray start` is a live process, not part of any manifest, so it would not survive a ray-client pod restart. If this needs to be durable, the cleaner fix is to have the client connect via Ray Client (`ray://ray-head:10001`) instead of the GCS address — but that would require changing the validation command, which the task fixed.
