**What happened:** The adversary had corrupted the `elastic-password` secret to `CORRUPTED_BY_ADVERSARY`, but the actual Elasticsearch password was still the previous value `Elastic1234!` (found via the `elastic-password-prev` ConfigMap). The rotation steps were:
1. Used the working old password to call `_security/user/elastic/_password` via `POST` on an ES node
2. Updated the `elastic-password` secret to the new value
3. Rolling-restarted `es-cluster` StatefulSet so pods picked up the new env var for their readiness probe
4. Patched the `auth-checker` deployment to use `https://` (the cluster enforces TLS) and restarted it
