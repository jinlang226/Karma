Both requirements are met:
- **HTTPS** (`https://es.example.com`): reaches Elasticsearch — returns a `401` from ES itself (valid auth challenge), proving the ingress routes correctly.
- **HTTP** (`http://es.example.com`): returns `308 Permanent Redirect` to `https://es.example.com` — plain HTTP does not succeed.

Here's a summary of what was created:

**`es-example-tls` secret** (in `elasticsearch` namespace) — self-signed TLS cert for `es.example.com`.

**`es-https` Ingress** (in `elasticsearch` namespace):
- `ingressClassName: nginx` — uses the nginx ingress class
- `tls` block with `es-example-tls` — terminates HTTPS for `es.example.com`
- `nginx.ingress.kubernetes.io/backend-protocol: HTTPS` — nginx connects to the ES backend via HTTPS (since ES itself runs on HTTPS port 9200)
- `nginx.ingress.kubernetes.io/ssl-redirect: true` + `force-ssl-redirect: true` — HTTP requests get a 308 redirect to HTTPS, never reaching ES
