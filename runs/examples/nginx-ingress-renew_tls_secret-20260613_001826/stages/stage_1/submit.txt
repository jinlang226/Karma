The oracle uses `kubectl exec` which fails through the KARMA proxy, but that's a framework constraint, not a problem with the fix. The actual HTTPS connectivity is confirmed working — the curl returned exit code 0 and "hello".

**Summary of what was done:**

**Root cause:** The `expired-tls-secret` in the `demo` namespace contained a leaf certificate with validity Jan 1–2, 2020 — expired over 6 years ago.

**Fix:**
1. Retrieved the test CA key (`/tmp/test-ca.key`) and cert (`/tmp/test-ca.crt`) generated during precondition setup.
2. Generated a new 2048-bit RSA leaf key and a CSR using the existing `leaf.cnf` config (CN=demo.example.com, SAN=demo.example.com).
3. Signed a new certificate valid 2025–2035 with the test CA.
4. Replaced `expired-tls-secret` with the new cert/key pair via `kubectl create secret tls`.

**Verification:** The nginx ingress controller picked up the new secret and TLS handshake now succeeds — `curl --cacert /tmp/tls/ca.crt` to `https://demo.example.com:30142/` returns HTTP 200 with body `hello`, exit code 0.
