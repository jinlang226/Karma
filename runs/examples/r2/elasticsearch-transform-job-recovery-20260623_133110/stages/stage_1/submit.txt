The transform is running and the checkpoint has advanced. Here's a summary of what was done and why:

**Root cause:** The `es-transform-config` ConfigMap had `node.roles: [ ingest ]` — the `transform` role was missing. This was the "capacity change" that broke things. Without a node holding the `transform` role, Elasticsearch rejects any attempt to create or run a transform job.

**Steps taken:**
1. **Patched the ConfigMap** to add `transform` to the node roles: `node.roles: [ ingest, transform ]`
2. **Scaled `es-transform` StatefulSet back to 1 replica** — the pod picks up the updated config on start
3. **Verified** the new node joined the cluster with the `transform` role
4. **Created** the `events-by-service` transform (it was never successfully created before)
5. **Started** the transform via `_start` API
6. **Confirmed** checkpoint advanced to 1 immediately, processing the backlogged `app-events` data into `app-events-rollup`
