Done. Here's what was wrong and what was fixed:

**Root cause:** The `health-overrides` ConfigMap contained a stale entry `mongodb-replica-1: legacyHealthPass123`. The health check script uses a per-pod override file if it exists, so `mongodb-replica-1` was attempting to authenticate as `health-user` with the old password `legacyHealthPass123` instead of the current password `healthpass123` from the `health-user-password` secret.

**Fix:** Removed the stale override entry from the `health-overrides` ConfigMap. The script now falls back to reading the password from the secret, which succeeds.

**Result:** All 3 pods are `1/1 Ready` with a stable 1 PRIMARY + 2 SECONDARY topology, no data loss.
