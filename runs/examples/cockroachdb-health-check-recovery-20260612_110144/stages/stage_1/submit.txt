The cluster is fully recovered. Summary of what happened and what was fixed:

**Root cause:** The StatefulSet command contained a deliberate fault — a shell conditional that bound `crdb-cluster-1`'s HTTP server to `127.0.0.1:8080` (loopback) instead of `0.0.0.0:8080`. This made the Kubernetes readiness probe (`/health?ready=1`) unreachable from outside the pod's loopback interface, so the probe failed and the pod was never marked Ready.

**Fix:** Patched the StatefulSet to remove the conditional and always use `--http-addr=0.0.0.0:8080` for all pods. Then manually deleted `crdb-cluster-1` to break the rolling-update deadlock (the update couldn't proceed because it waits for the pod to become Ready, but the pod needed the update to become Ready).

**Verification:**
- All 3 pods: `1/1 Running`, 0 restarts
- StatefulSet: `3/3 READY`
- No Unhealthy events for any pod since the recreation
