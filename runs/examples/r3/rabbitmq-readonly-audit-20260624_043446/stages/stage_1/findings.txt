SECTION 2 — TLS / TRANSPORT SECURITY
--------------------------------------
AMQP listener:      TCP port 5672 (plaintext)
Management listener: TCP port 15672 (plaintext HTTP)
SSL listeners:      NONE configured
TLS options:        ssl_options = [] (empty — TLS disabled)
Management HTTPS:   NOT configured

FINDING [CRITICAL] — AMQP traffic (port 5672) is transmitted in plaintext.
Credentials and message payloads are exposed to anyone with network access to
the cluster or underlying node network. mTLS should be enabled on the AMQP
listener.

FINDING [CRITICAL] — The RabbitMQ Management API (port 15672) uses plain HTTP.
Admin credentials are transmitted in cleartext on every API call and browser
login. TLS must be enabled for the management interface.

Both services (rabbitmq, rabbitmq-headless) are ClusterIP only (no NodePort/
LoadBalancer), so exposure is currently limited to in-cluster traffic. However,
any compromised pod or service-account token can reach these plaintext ports.

---------------------------------

Permissions on vhost "/":
  admin       configure=.* write=.* read=.*

Permissions on vhost "/app":
  admin       configure=.* write=.* read=.*
  app-user    configure=.* write=.* read=.*

FINDING [HIGH] — The "guest" default account is accessible from non-loopback
addresses. Config file sets "loopback_users.guest = false", overriding the
RabbitMQ default that restricts guest to localhost only. If the guest account
password has not been changed from the default ("guest"), this constitutes a
trivially exploitable remote credential. The guest user should either be deleted
or have loopback restriction re-enabled.

FINDING [HIGH] — "app-user" has configure=.* (full topology permissions) on
/app. Application service accounts should be granted only the permissions they
require. If app-user only publishes/consumes, configure permissions should be
restricted (e.g., configure="" write="app-queue" read="app-queue").

FINDING [MEDIUM] — "app-user" has no tags, meaning it has no access to the
management UI or monitoring endpoints. This is appropriate for an application
credential, but note that there is no dedicated monitoring/readonly user for
operations visibility without using the admin credential.

Password hashing module: rabbit_password_hashing_sha256 (SHA-256).
FINDING [LOW] — RabbitMQ supports SHA-512 hashing
(rabbit_password_hashing_sha512) which provides stronger password storage.
Current configuration uses SHA-256.

No user-level connection or channel limits are configured.
FINDING [LOW] — No per-user connection or channel limits are set. A
misconfigured or malicious client could open unlimited connections.

Vhost "/app": No policies defined.

No operator policies configured on any vhost.

FINDING [HIGH] — No HA/quorum or mirroring policies are defined. The only
queue in the cluster (app-queue) is a classic queue with no redundancy policy,
meaning it has a single master and no replicas. If rabbitmq-0 (the queue
