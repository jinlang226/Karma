The ETL pipeline is now **Complete**. Here's a summary of the diagnosis and fix:

---

**Root Cause (two-part)**

| Issue | Detail |
|---|---|
| Insufficient executor memory | `spark.executor.memory=256m` (268 MB) is below Spark's minimum of ~450 MB — the JVM crashed before any work started |
| Data skew | `user_id=999` holds ~50% of all records (500x skew ratio); the Stage 3 `GROUP BY user_id` funnels all that data into a single executor partition |

**Fix applied** (`etl-job.yaml`, line 32–33):
- Raised `spark.executor.memory` from `256m` → `1g` (clears the hard minimum and provides headroom for the skewed partition)
- Enabled `spark.sql.adaptive.enabled=true` + `spark.sql.adaptive.skewJoin.enabled=true` — AQE will split the oversized `user_id=999` partition into smaller chunks at runtime, preventing any single executor from holding 50% of the dataset in memory

**Result**: Job status `Complete 1/1` in 49 seconds, ending with `spark_etl_pipeline_verified`.
