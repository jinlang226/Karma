ConfigMap `rollback-rehearsal` is created and verified. The live cluster is unchanged.

**What was found and what the script reverts:**

| # | Type | Setting | Current (non-default) | Will revert to |
|---|------|---------|-----------------------|----------------|
| 1 | Cluster setting | `kv.snapshot_rebalance.max_rate` | 1.0 MiB | 32 MiB (default) |
| 2 | Cluster setting | `diagnostics.reporting.enabled` | true | false (default) |
| 3 | Zone config | `RANGE default` `gc.ttlseconds` | 14400 s (4 h) | 90000 s (25 h, default) |

**Intentionally excluded from the script:**
- `cluster.secret` — auto-generated cluster identity UUID; resetting it would break inter-node replication.
- `version = 24.1` — the cluster's upgrade level; CockroachDB does not support version downgrade.

All other zone entries (`DATABASE system`, `RANGE meta`, `RANGE liveness`, etc.) match the system-internal defaults CockroachDB sets on every new cluster and were not touched.

The script uses `RESET CLUSTER SETTING` for the two cluster settings (idiomatic — removes the explicit override and lets the built-in default apply), and an explicit `CONFIGURE ZONE USING gc.ttlseconds = 90000` for the zone config. It also runs post-rollback verification queries so the operator can confirm the state after execution.
