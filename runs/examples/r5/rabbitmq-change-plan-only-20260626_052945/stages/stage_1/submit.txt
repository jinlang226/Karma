The `change-plan` ConfigMap is created with a 10KB `plan.md`. Here's a summary of what was found and planned:

**Current cluster state:**
- RabbitMQ **3.12.14** on Erlang 25.3.2.15 — marked **out of support** by the project itself
- 3 healthy disk nodes, no alarms, no network partitions
- 2 vhosts: `/` (default) and `/app`
- 2 users: `admin` (administrator) and `app-user` (no tags, full access to `/app` only)
- No policies defined on either vhost
- One queue: `app-queue` in `/app` — type **classic**, durable, no arguments

**Key issues identified in the plan:**
1. **Critical:** 3.12.x is end-of-life — needs upgrade to 3.13 then 4.0
2. **High:** `app-queue` is a classic queue; classic queues are removed in 4.0 and must be migrated to quorum queues first
3. **Medium:** No HA policy exists — `app-queue` has no replication
4. **Low:** `queue_master_locator = min-masters` is deprecated/removed in 4.0; replace with `queue_leader_locator = balanced`

**Planned phases:** queue migration → policy → config fix → rolling 3.12→3.13 upgrade → rolling 3.13→4.0 upgrade → validation. No changes were applied to the live cluster.
