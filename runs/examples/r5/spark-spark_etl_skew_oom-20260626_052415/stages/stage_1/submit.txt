## Diagnosis & Resolution Summary

### Root Cause: Data Skew → Stage 3 OOM

**Data distribution:**
- Total records: 100,000 across 1,000 distinct user IDs
- `user_id 999` holds **50,000 records (50% of all data)**
- **Skew ratio: 500x** (far above the 10x alert threshold)

**Why it causes OOM:** Stage 3 executes `GROUP BY user_id`. With HashAggregate partitioning, all 50,000 records for `user_id 999` land on a **single executor partition**. With only 256MB executor memory, that partition cannot hold the data in memory → `OutOfMemoryError`.

---

### Fix Applied (Options B + C combined)

The `etl-job.yaml` has the remediation in place:

```yaml
/opt/spark/bin/spark-submit \
  --master local[2] \
  --conf spark.executor.memory=1g \                    # ← raised from 256m (Option B)
  --conf spark.sql.adaptive.enabled=true \             # ← AQE enabled (Option C)
  --conf spark.sql.adaptive.skewJoin.enabled=true \    # ← skew join handling (Option C)
  ...
```

- **Option B:** Executor memory raised from 256m → 1g (4x), giving the skewed partition headroom
- **Option C:** AQE skew join splits oversized partitions automatically, preventing any single executor from being overwhelmed
