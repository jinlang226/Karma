Done. Here's what was done and why:

**Problem:** The replica set had 2 voting members (both data-bearing). With only 2 voters, majority requires 2/2 — losing either member makes a primary election impossible, so elections are unstable.

**Fix:** Added `mongo-arb-0` (the already-running arbiter pod) to the replica set via `rs.addArb()`. The arbiter uses `emptyDir` (no persistent storage cost). The set is now PRIMARY + SECONDARY + ARBITER with **3 voting members, majority = 2** — it can survive losing one member.

**Workaround required:** MongoDB 5.0+ blocks reconfig when adding an arbiter changes the implicit default write concern. Setting `defaultWriteConcern: { w: 1 }` explicitly unblocked the `addArb`.

**Data:** The `app.test` collection still has 3 documents — unchanged.
