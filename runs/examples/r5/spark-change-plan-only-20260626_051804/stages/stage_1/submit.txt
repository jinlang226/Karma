The ConfigMap is created and the live cluster is untouched. Here's a summary of what was done:

**Inspection findings:**
- All workloads (master, worker, job) run `apache/spark:3.5.3` with identical resource limits (500m CPU / 1Gi RAM, 200m/512Mi requests)
- The spark-pi Job uses `--master local[2]` — it runs a local Spark context and never talks to the standing master/worker Deployment
- Worker has `SPARK_WORKER_MEMORY=1G` equal to its container memory limit — OOM risk
- RBAC grants `delete` on pods (overly broad)
- No liveness/readiness probes, no ResourceQuota, single worker replica

**Plan written in `change-plan` ConfigMap (`plan.md` key) covers 8 ordered steps:**
1. RBAC tightening (remove `delete pods` verb)
2. Add ResourceQuota (cap at 3 CPU / 4Gi)
3. Fix worker memory headroom (1G → 768m)
4. Add HTTP liveness/readiness probes to master and worker
5. Scale worker to 2 replicas
6. Expose worker WebUI via Service
7. Switch job from local mode to cluster mode (`spark://spark-master:7077`)
8. Upgrade image to 3.5.4 with digest pinning

Each step includes the exact commands, verification checks, and a rollback strategy.
