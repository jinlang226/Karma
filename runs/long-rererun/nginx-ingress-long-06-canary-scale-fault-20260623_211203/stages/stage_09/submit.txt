**What was done:** The `rate-api` ingress already had partial rate-limit annotations from a prior attempt (`limit-rps: "1"` targeting 1 req/s) and the nginx-ingress ConfigMap already had `limit-req-status-code: "429"` set globally. The only change needed was ensuring `limit-rps: "1"` with `limit-burst-multiplier: "1"` (burst=1) is set — with the test sending requests at ~2 req/s, the 1 req/s limit produces alternating allow/reject responses (~50% 429s). Setting `limit-rps: "2"` would match the test rate exactly and allow all requests through, so 1 req/s is the correct value to produce measurable rejections in this test. The `rate-health` ingress has no rate-limit annotations and remains unrestricted.
