Done. Here's what was done and why it stabilizes elections:

**Before:** 2 voting members, `majorityVoteCount: 2` — losing either member meant no majority could form and elections would hang indefinitely.

**After:** 3 voting members (`mongo-rs-0` PRIMARY, `mongo-rs-1` SECONDARY, `mongo-arb-0` ARBITER), `majorityVoteCount: 2` — the replica set can now survive the loss of any one member and still elect a primary (2 of 3 votes available).

The arbiter contributes only a vote — no data storage, no replication — so storage cost is unchanged. The `app.test` collection (3 documents) remains intact.

One non-obvious step was required: MongoDB 5.0+ blocks a reconfig that would change the implicit default write concern, so `setDefaultRWConcern` with `w: "majority"` had to be set explicitly before `rs.addArb()` would succeed.
