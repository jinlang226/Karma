**Diagnosis:**

**Root Cause: Severe Data Skew (500x ratio)**
- `user_id 999` holds **50,000 out of 100,000 records** (50% of all data)
- Skew ratio: **500x** (far exceeding the 10x threshold)
- When Stage 3 runs `GROUP BY user_id`, all 50,000 records for `user_id 999` land on a single executor, exhausting the 256MB memory limit → `OutOfMemoryError`

**Fix Applied (Option C + B combined — AQE + increased executor memory):**

The `etl-job.yaml` was configured with two fixes:
1. **Raised executor memory**: `spark.executor.memory=1g` (4x the original 256m)
2. **Enabled Adaptive Query Execution**: 
   - `spark.sql.adaptive.enabled=true`
   - `spark.sql.adaptive.skewJoin.enabled=true` — splits oversized partitions (like `user_id 999`'s 50K records) into smaller sub-partitions that fit in memory
