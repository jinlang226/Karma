All goals achieved. Here's a summary of what was done and why:

**Root cause:** The ConfigMap `es-config` was missing `cluster.initial_master_nodes`. In Elasticsearch 7.x, this setting is required for the very first cluster bootstrap — without it, nodes enter discovery but can never elect a master because they don't know which nodes form the initial quorum.

**Fix in three steps:**

1. **Added `cluster.initial_master_nodes`** to the ConfigMap listing all three node names (`es-cluster-0`, `es-cluster-1`, `es-cluster-2`).

2. **Restarted all pods** so they loaded the new config. The cluster bootstrapped, a master (`es-cluster-1`) was elected, and all pods became 1/1 Ready. Status: **green, 3 nodes**.

3. **Removed `cluster.initial_master_nodes`** from the ConfigMap — this is a bootstrap-only setting. Once the cluster state is persisted to disk, ES ignores it on restarts, but leaving it in could cause confusion or warnings. The clean config has only `discovery.seed_hosts` for ongoing peer discovery.

4. **Rolling restart completed successfully** — all 3 pods restarted in sequence without `cluster.initial_master_nodes`, the cluster remained green throughout (master re-elected to `es-cluster-2`), confirming the fix is safe for future restarts.
