## Summary

The cluster was upgraded from **RabbitMQ 3.9.29 → 4.1.8** using only native Kubernetes resources (a rolling `kubectl set image` on the existing StatefulSet, PVCs retained throughout).

**Why stepwise:** RabbitMQ does not support skipping minor versions, and the 3.x→4.0 major boundary requires all stable feature flags to be enabled first. A direct 3.9→4.1 jump would have failed on schema migration and feature-flag requirements. I followed the supported chain, enabling all stable feature flags at each step:

`3.9 → 3.10 → 3.11 → 3.12 → 3.13 → 4.0 → 4.1`

Each step was a RollingUpdate of the StatefulSet (pods restarted one at a time, each reusing its `data-rabbitmq-N` PVC to migrate its on-disk Mnesia store), with confirmation that all 3 nodes reached the target version and the cluster stayed healthy before proceeding.

**Final state:**
- All 3 nodes (`rabbitmq-0/1/2`) running **RabbitMQ 4.1.8** on Erlang 27.
- Cluster healthy: 3 running nodes, no alarms, no network partitions.
- Data preserved: `/app` vhost present, `app-queue` (classic, durable) intact with **5 messages**.
- No CRDs, Operators, or third-party controllers used — only the native StatefulSet/Service/PVC/Secret/ConfigMap stack.

One note: the experimental `khepri_db` feature flag was intentionally left disabled (Mnesia remains the default metadata store in 4.0/4.1 and Khepri is not required for the upgrade).
