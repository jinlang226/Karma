## Summary

**Diagnosis:** The 3 CockroachDB pods were running with a correct `--join`/`--advertise-host` configuration (the headless service even has `publishNotReadyAddresses: true`, so peer DNS resolves before readiness). The "join/advertise errors" were transient — pods were still in `ContainerCreating` and DNS for peer pods hadn't resolved yet. Once all three were running, the logs showed the real blocker:

```
awaiting `cockroach init` or join with an already initialized node
‹crdb-cluster-1...› is itself waiting for init, will retry
```

All nodes were reachable but **none was bootstrapped** — the cluster had simply never been initialized.

**Fix (used existing resources / in-cluster tooling, no recreation):**
```bash
kubectl -n cockroachdb exec crdb-cluster-0 -- ./cockroach init --insecure --host=localhost:26257
# → Cluster successfully initialized
```
