The CockroachDB cluster is fully deployed and healthy.

## Summary

A 3-node CockroachDB cluster is running in the `cockroachdb` namespace using only native Kubernetes resources (manifest: `crdb.yaml`).

**Resources created:**
- **Headless Service** `crdb-cluster` (`clusterIP: None`, `publishNotReadyAddresses: true`) — provides stable pod DNS for peering. Exposes grpc/26257 + http/8080.
- **Public Service** `crdb-cluster-public` (ClusterIP) — client/UI entrypoint. Same two ports.
- **StatefulSet** `crdb-cluster` — 3 replicas, image `cockroachdb/cockroach:v24.1.0`, each pod advertises its own DNS (`$(hostname).crdb-cluster.cockroachdb.svc.cluster.local`) and joins all three nodes via `--join`. Insecure mode (`--insecure`, no TLS).
- **VolumeClaimTemplate** `datadir` — 10Gi RWO per pod (all 3 PVCs Bound).
- **PodDisruptionBudget** `crdb-cluster-pdb` — `maxUnavailable: 1`, so voluntary disruptions can only evict one node at a time, preserving quorum (2 of 3 always available). Plus soft pod anti-affinity to spread across nodes.

**Constraints honored:** existing `crdb-cluster-sa` ServiceAccount reused (no new RBAC), no CRDs/operator, exact names and ports as specified.

**Verification:**
- StatefulSet `3/3` Ready; all pods `1/1 Running`.
- One-time `cockroach init` performed (required to bootstrap a fresh cluster — without it pod-0 stays NotReady and the OrderedReady StatefulSet won't progress).
- `cockroach node status` shows all 3 nodes `is_available=true`, `is_live=true`.
