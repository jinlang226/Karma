## What was wrong
The replica set had only **2 voting members** (`mongo-rs-0`, `mongo-rs-1`) — an even vote count, which makes elections unstable (no member can reliably reach a strict majority). The arbiter pod `mongo-arb-0` was running but had **never been added** to the replica set config, so it wasn't voting.

## What I did
1. Confirmed the arbiter pod was reachable and the config had only 2 data members.
2. `rs.addArb()` initially failed because adding an arbiter to a PSA topology changes the implicit default write concern (MongoDB 5.0+ safeguard). I pinned it explicitly with `setDefaultRWConcern: {w: "majority"}` to preserve write durability.
3. Added the arbiter as a voting, `arbiterOnly` member.

## Result
- **3 voting members** now (2 data + 1 arbiter), giving an odd vote count → stable elections. State: `PRIMARY` / `SECONDARY` / `ARBITER`, all healthy.
- **No data-bearing member added** — the arbiter holds no data (`arbiterOnly: true`).
- **No new storage** — still exactly 2 PVCs (the arbiter has no PVC).
- **Data intact** — `app.test` still holds its 3 documents.
