Everything is working. Here's what was done:

**Root cause:** The `prometheus-config` ConfigMap had no RabbitMQ scrape job — Prometheus only scraped itself.

**Fix (two changes to the ConfigMap):**

1. **Added a `rabbitmq` scrape job** targeting all 3 pods individually via the headless service DNS (`rabbitmq-{0,1,2}.rabbitmq-headless:15692`). The headless service resolves each ordinal pod by name, so no load-balancing hides individual targets.

2. **Added a `rules.yml` recording rule** (`rabbitmq_up = up{job="rabbitmq"}`) because the RabbitMQ built-in Prometheus plugin (port 15692) doesn't emit a `rabbitmq_up` metric — that name only exists in the old standalone exporter. The recording rule derives it from Prometheus's synthetic `up` metric, which is 1 for every healthy scrape target.

Results:
- 3 RabbitMQ targets: all **UP**
- `rabbitmq_up` query: **3 samples** (all = 1)
- All 3 RabbitMQ pods: **Running**, no restarts
