## What I did

The Elasticsearch HTTP API on `es-http:9200` already has TLS + security enabled (it speaks HTTPS and returns `401` for unauthenticated requests). I exposed it through ingress-nginx:

1. **TLS secret** — generated a self-signed cert for `CN=es.example.com` and created secret `es-tls` in the `elasticsearch` namespace.
2. **Ingress** `es-ingress` (`ingressClassName: nginx`) for host `es.example.com` → `es-http:9200`, with:
   - `nginx.ingress.kubernetes.io/backend-protocol: HTTPS` — required because ES's own endpoint is HTTPS, not plain HTTP.
   - `nginx.ingress.kubernetes.io/force-ssl-redirect: "true"` — blocks plain HTTP.
   - A `tls:` block referencing `es-tls`.

## Verification
- **HTTPS** `https://es.example.com/` → `HTTP 401` — the request reaches Elasticsearch (auth challenge), so the endpoint is reachable as required.
- **Plain HTTP** `http://es.example.com/` → `HTTP 308` redirect, never serves data — plain HTTP does not succeed.
- All Elasticsearch pods (`es-0/1/2` + others) and the ingress-nginx controller are `Running` — cluster healthy.

Constraints respected: no operators/CRDs installed, used the `nginx` ingress class.

Note: the cert is self-signed, so external clients need `-k` or to trust the cert / use a real CA. The reachability and HTTP-blocking requirements are both satisfied.
