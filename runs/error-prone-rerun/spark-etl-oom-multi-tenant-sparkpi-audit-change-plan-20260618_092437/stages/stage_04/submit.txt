## What I found

**Correctly configured (PASS)**
1. **Dedicated ServiceAccount** — all workloads run under `spark-pi`, not `default`.
2. **Namespace-scoped RBAC** — a `Role` + `RoleBinding` (not a ClusterRole), binding correctly targets the `spark-pi` SA.
3. **Resource governance** — every workload (master, worker, job) sets both requests (200m/512Mi) and limits (500m/1Gi).
4. **Image tag pinning** — all use `apache/spark:3.5.3`, an explicit version (not `:latest`).

**Gaps (FAIL/WARN)**
- **F1 [HIGH]** — `securityContext: {}` everywhere. No `runAsNonRoot`, `allowPrivilegeEscalation: false`, `readOnlyRootFilesystem`, or dropped capabilities → containers run as root with default caps.
- **F2 [MEDIUM]** — RBAC over-permission **and config drift**: the live `spark-pi-role` grants write verbs on `pods`, but `pods` is **not** in the `last-applied-configuration` — it was added out-of-band. Broad `delete/update/patch` verbs exceed what a batch job needs.
- **F3 [MEDIUM]** — No NetworkPolicy in the namespace (default-allow ingress/egress).
- **F4 [LOW]** — Image provenance not cryptographically pinned: mutable tag from public Docker Hub, no `@sha256` digest.

Per the constraints, I made no remediation — the cluster remains exactly as found.
