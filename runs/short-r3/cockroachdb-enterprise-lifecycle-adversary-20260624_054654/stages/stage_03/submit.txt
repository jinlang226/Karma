Both goals are achieved:

1. **Insecure access rejected** — `./cockroach sql --insecure` returns `server error: node is running secure mode, SSL connection required` (exit code 1).

2. **Secure access succeeds** — `./cockroach sql --certs-dir=/cockroach/cockroach-certs -e "SELECT 1;"` returns `1` (exit code 0).

**What was done:**

- Generated a CA, node certificate (with SANs for all three pod DNS names, the headless service, and the public service), and a root client certificate — all using `cockroach cert` on the existing pod.
- Created a Kubernetes secret `cockroachdb-certs` containing all five files (`ca.crt`, `node.crt`, `node.key`, `client.root.crt`, `client.root.key`) with `defaultMode: 0600`.
- Patched the StatefulSet: replaced `--insecure` with `--certs-dir=/cockroach/cockroach-certs`, mounted the cert secret at `/cockroach/cockroach-certs`, switched liveness/readiness probes to HTTPS, and updated `COCKROACH_CHANNEL` to `kubernetes-secure`.
- Forced simultaneous restart of all three pods (required because CockroachDB cannot run mixed insecure/secure nodes in the same cluster).
