The CockroachDB cluster is fully deployed and healthy. Here's a summary of what was created:

**Resources deployed:**
- **Headless Service** `crdb-cluster` — DNS-based peer discovery on ports 26257 (grpc) and 8080 (http)
- **Public Service** `crdb-cluster-public` — ClusterIP service for client access on the same ports
- **StatefulSet** `crdb-cluster` — 3/3 replicas ready, using `podManagementPolicy: Parallel`
- **PVCs** — 3× 10Gi `datadir` volumes, all Bound
- **PodDisruptionBudget** `crdb-cluster-pdb` — `minAvailable: 2`, allowing 1 voluntary disruption at a time (preserves quorum)

**Cluster status:** All 3 nodes report `is_available=true` and `is_live=true`, advertising via their full DNS names (`crdb-cluster-{0,1,2}.crdb-cluster.cockroachdb.svc.cluster.local:26257`).

The `Parallel` pod management policy was required because CockroachDB's `--join` bootstrapping is a chicken-and-egg: all nodes must be Running before `cockroach init` can form the cluster, so the default `OrderedReady` policy would deadlock waiting for pod-0's readiness probe.
