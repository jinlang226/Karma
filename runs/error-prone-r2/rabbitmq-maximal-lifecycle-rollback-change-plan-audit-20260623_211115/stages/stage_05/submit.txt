Done. Here's a summary of what was changed and why:

**Root cause:** The `prometheus-config` ConfigMap only had a self-scrape job (`localhost:9090`) — no RabbitMQ job at all.

**Fix — one ConfigMap update, two additions:**

1. **Added a `rabbitmq` scrape job** targeting each pod individually via the headless service DNS (`rabbitmq-{0,1,2}.rabbitmq-headless.<ns>.svc.cluster.local:15692`). Using the headless service ensures Prometheus talks to each pod separately and shows 3 distinct targets rather than a load-balanced single target through the ClusterIP service.

2. **Added a recording rule `rabbitmq_up = up{job="rabbitmq"}`** in a `rabbitmq_rules.yml` key. RabbitMQ's built-in Prometheus plugin (port 15692) doesn't expose `rabbitmq_up` — that's only from the community `rabbitmq_exporter`. Prometheus auto-generates an `up` metric per target (1 = reachable, 0 = unreachable); the recording rule aliases it to the expected name. No external exporter, CRD, or operator needed.

Results:
- 3 RabbitMQ targets `health: "up"` in Prometheus
- `rabbitmq_up` returns 3 samples, all value `"1"`
- All 3 RabbitMQ pods still `1/1 Running`, no restarts
