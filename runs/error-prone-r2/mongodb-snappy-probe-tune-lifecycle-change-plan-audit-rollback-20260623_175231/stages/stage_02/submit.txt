The replica set has been successfully scaled to 5 members. Here's a summary of what was done and what issues were encountered:

**Result:** All 5 members are healthy — 1 PRIMARY (`mongodb-replica-1`), 4 SECONDARY (`replica-0`, `2`, `3`, `4`). No data loss, no loss of availability (the 3-member primary/secondaries remained up throughout).

**What I did:**
1. **Scaled the StatefulSet** from 3 → 5 replicas to provision pods 3 and 4.
2. **Cleaned stale/corrupt PVCs** — the new pods crashed due to WiredTiger data from prior failed attempts missing the FCV document. I ran cleanup pods to wipe the data directories.
3. **Added a `repair-if-unclean` init container** to the StatefulSet that runs `mongod --repair` if `mongod.lock` is non-empty, preventing the FCV crash loop on subsequent restarts.
4. **Fixed the liveness/readiness probes** — they originally used `admin-user:password` auth, which failed on fresh nodes until user data is replicated from the primary. Changed to unauthenticated `ping` (which MongoDB allows without auth) so new nodes stay up long enough to complete initial sync.
5. **Called `rs.add()`** on the primary to add `mongodb-replica-3` and `mongodb-replica-4` to the replica set.
