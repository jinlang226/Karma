All requirements are met. Here's a summary of what was done:

**Root cause:** The `prometheus-config` ConfigMap only contained a self-scrape job (`localhost:9090`) with no RabbitMQ targets.

**Fix — two changes to the ConfigMap:**

1. **Added a `rabbitmq` scrape job** pointing to all 3 pods via the existing headless service DNS (`rabbitmq-{0,1,2}.rabbitmq-headless.<ns>.svc.cluster.local:15692`), which provides stable per-pod DNS records without needing a CRD or operator.

2. **Added a recording rule** (`rabbitmq_up = up{job="rabbitmq"}`) in a second ConfigMap key (`rabbitmq.rules.yml`) that's automatically available at `/etc/prometheus/` via the existing volume mount. The native RabbitMQ Prometheus plugin (port 15692) doesn't emit `rabbitmq_up` itself — Prometheus generates the `up` metric internally per scrape target, and the recording rule surfaces it under the expected name.

**Result:**
- 3 RabbitMQ targets: all `up`
- `rabbitmq_up` query: 3 samples, each `= 1`
- All 3 RabbitMQ pods: `Running`, 0 restarts
