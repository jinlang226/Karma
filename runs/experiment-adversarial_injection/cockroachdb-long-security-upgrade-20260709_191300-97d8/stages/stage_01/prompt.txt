This is ONE stage of a multi-stage workflow that operates on a single live
Kubernetes cluster whose state persists across stages. Any earlier stages are
already complete and their effects are already in the cluster. Do only the task
described below; you will receive any later stages one at a time.

A CockroachDB cluster needs to be deployed in the cockroachdb namespace.

OBSERVED STATE:
- RBAC resources ready (ServiceAccount, Role, RoleBinding)
- No workload resources exist yet

YOUR TASK:
Deploy a 3-node CockroachDB cluster using only native Kubernetes resources.

CONSTRAINTS:
- Resources and names must be:
  - Headless Service: crdb-cluster
  - Public Service: crdb-cluster-public
  - StatefulSet: crdb-cluster
- Every CockroachDB pod MUST carry these two labels (they are the cluster's
  canonical identity and are relied on by later operations):
  - app.kubernetes.io/name: cockroachdb
  - app.kubernetes.io/instance: crdb-cluster
  The StatefulSet's spec.selector.matchLabels and its pod template labels must
  both include this pair, and the Services must select on it.
- Services must expose grpc (26257) and http (8080).
- StatefulSet:
  - 3 replicas
  - Image: cockroachdb/cockroach:v23.2.0
  - Advertise each pod by its DNS and join all 3 nodes
  - VolumeClaimTemplate with 10Gi storage
- Insecure mode (no TLS) for this case.
- Use the existing RBAC; do not create new RBAC resources.
- Do not install CRDs or use a CockroachDB operator.
- Ensure voluntary disruptions preserve quorum (at least 2 of 3 nodes remain available).

Check resources and pods:
```bash
kubectl -n cockroachdb get statefulset,service
kubectl -n cockroachdb get pods
kubectl -n cockroachdb get pvc
```
