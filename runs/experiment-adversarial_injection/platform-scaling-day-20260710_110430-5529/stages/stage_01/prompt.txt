This is ONE stage of a multi-stage workflow that operates on a single live
Kubernetes cluster whose state persists across stages. Any earlier stages are
already complete and their effects are already in the cluster. Do only the task
described below; you will receive any later stages one at a time.

A 3-node MongoDB replica set is not deployed in the mongodb namespace.

Task:
- Deploy a 3-member replica set with authentication enabled. Back it with a
  StatefulSet named `mongodb-replica` (pods `mongodb-replica-0..2`).
- Name the replica set itself `mongodb-replica` (the mongod `replSetName` /
  `rs.initiate` `_id` must be exactly `mongodb-replica`, not the `rs0` default).
- Create a headless Service named `mongodb-replica-svc` (clusterIP: None) that
  governs the StatefulSet, so each member is reachable at its stable DNS name
  `mongodb-replica-<n>.mongodb-replica-svc.mongodb.svc.cluster.local`.
- Create an admin user and an application user.
- Ensure the replica set is healthy (PRIMARY + SECONDARY members).
- Confirm data can be written and read from the application database.

Credentials:
- Admin user: admin-user / admin123password
- App user: app-user / app123password
- Store the passwords in Secrets:
  * admin-user-password (key: password)
  * app-user-password (key: password)

Notes:
- Use a keyfile for internal replica set authentication.
- No operators or CRDs.
