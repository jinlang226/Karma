Fixed. The root cause was a RabbitMQ permission gap: `app-user` had `write`/`read` but empty `configure` permission on the `/app` vhost, so it couldn't declare `app-queue` via the management API. I granted `configure` (and reaffirmed write/read) with `rabbitmqctl set_permissions -p /app app-user ".*" ".*" ".*"`, then recycled the producer pod.

Current state:
- `app-producer` is `1/1 Running`, 0 restarts, no more crash loop.
- `/app` vhost has `app-queue` of type `classic` with messages actively accumulating (10 and growing).
- All 3 RabbitMQ nodes (`rabbitmq-0/1/2`) are up and in cluster status as disk+running nodes — no data loss, no CRDs/operators touched.
