This is ONE stage of a multi-stage workflow that operates on a single live
Kubernetes cluster whose state persists across stages. Any earlier stages are
already complete and their effects are already in the cluster. Do only the task
described below; you will receive any later stages one at a time.

An ETL pipeline is failing with a memory error in Stage 3.
YOU must diagnose the root cause and fix the problem.

PROBLEM DESCRIPTION:
- The ETL pipeline has 3 stages:
  - Stage 1: Load raw data (events, users, products)
  - Stage 2: Clean and transform data
  - Stage 3: Aggregate by user_id  <-- FAILS HERE with a memory error
- The pipeline keeps failing in Stage 3 with an insufficient-memory error
  (the JVM heap is far below what Spark needs)
- Initial investigation shows the job is configured with only 256m of
  driver/executor memory

YOUR TASK:

1. COLLECT FAILURE INFORMATION
   - Check the ETL job status and error message
   - Review driver and executor logs
   - Look for memory errors (OutOfMemory / "must be at least ... increase
     heap size" messages)

   ```bash
   # Check ETL job status
   kubectl get job etl-job -n spark-etl

   # Get job pod logs
   kubectl logs -n spark-etl -l app=etl-job --tail=100

   # Check for failed pods
   kubectl get pods -n spark-etl
   ```

2. DIAGNOSE ROOT CAUSE
   - Analyze the data distribution
   - Look for data skew (one user_id has disproportionate records)
   - Calculate the skew ratio

   ```bash
   # Check data skew info
   kubectl exec -n spark-etl -l app=data-generator -- cat /data/skew_info.txt

   # Check diagnostic server logs
   kubectl logs -n spark-etl -l app=diagnostic-server --tail=50
   ```

   Expected finding: user_id 999 has ~50% of all records (skew ratio > 10x)

3. FIX THE PROBLEM
   The job's 256m heap is below what Spark needs, so the required fix is to
   raise the job's memory:
   ```bash
   # Delete failed job
   kubectl delete job etl-job -n spark-etl

   # Create a new job with increased memory (edit and apply):
   # raise the 256m memory settings to 1g — both --driver-memory and
   # spark.executor.memory (in local[2] mode the executor runs inside the
   # driver JVM) — in etl-job.yaml
   kubectl apply -f cases/spark/spark_etl_skew_oom/resource/etl-job.yaml
   ```

   Optionally, for the underlying data skew you may ALSO scale/upsize the
   workers or enable AQE skew handling on the job:
   ```bash
   # Scale workers for more parallelism
   kubectl scale deployment spark-worker -n spark-etl --replicas=4
   kubectl set env deployment/spark-worker -n spark-etl SPARK_WORKER_MEMORY=2G
   kubectl rollout status deployment/spark-worker -n spark-etl

   # And/or add AQE config to the job:
   # --conf spark.sql.adaptive.enabled=true
   # --conf spark.sql.adaptive.skewJoin.enabled=true
   ```

4. RE-RUN THE PIPELINE
   After applying the fix:
   ```bash
   # Re-apply the ETL job
   kubectl apply -f cases/spark/spark_etl_skew_oom/resource/etl-job.yaml

   # Watch job status
   kubectl get job etl-job -n spark-etl -w
   ```

5. VERIFY SUCCESS
   ```bash
   # Check job completed
   kubectl get job etl-job -n spark-etl

   # Check job logs
   kubectl logs -n spark-etl -l app=etl-job --tail=50
   ```

SUCCESS CRITERIA:
- Identified the memory failure in Stage 3
- Diagnosed data skew (user_id 999 with >10x ratio)
- Applied a fix (raised the job's memory above 256m; optionally also more
  workers or AQE)
- ETL job completed successfully
