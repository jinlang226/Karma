The `change-plan` ConfigMap is in the `demo` namespace with a 12,318-byte `plan.md`. The cluster is untouched.

Here's what the plan covers, based on what I found in the cluster:

**Current state discovered:**
- Controller v1.14.1, single replica, RollingUpdate with `maxUnavailable=1`
- `ingress-nginx-controller` ConfigMap has no custom data (all defaults)
- 7 Ingress resources across 4 virtual hosts

**Three active issues documented as pre-upgrade blockers:**
1. **`terminationGracePeriodSeconds: 0`** — the `/wait-shutdown` preStop hook gets no time to run, causing connection resets on every rolling restart
2. **Duplicate `--watch-ingress-without-class` flags** — both `=true` (original apply) and `=false` (later patch) are in the args list; the conflict silently resolves in favor of the last value but will flip if the original manifest is re-applied
3. **`canary-alt` missing base Ingress** — this canary Ingress targets `canary-alt.example.com` but no base Ingress exists for that hostname, generating a repeated reconcile error in every controller log

**Plan sections:** pre-upgrade checklist → ConfigMap settings to carry forward → image upgrade via `kubectl set image` → post-upgrade verification suite (rate limiting, canary routing, path rewrite) → rollback procedure.
