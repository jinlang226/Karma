The ConfigMap `config-audit` is created in the `spark-pi` namespace with a populated `findings.txt` key. The cluster was not modified — only the new ConfigMap was added.

Here's a summary of the audit findings:

**5 PASS (compliant):**
- Dedicated `spark-pi` ServiceAccount used by all workloads (no `default` SA usage)
- RBAC scoped as a namespace Role, not ClusterRole
- Resource requests and limits defined on all containers (Burstable QoS)
- Image pinned to `apache/spark:3.5.3` (not `:latest`)
- Job configured with `backoffLimit: 0` / `restartPolicy: Never` (fail-fast)

**3 FAIL (HIGH severity):**
1. **RBAC over-permission (GAP-1):** `spark-pi-role` grants `pods` write access (create/delete/update/patch) — this was added post-initial-creation per the `rollback-rehearsal` ConfigMap. SparkPi with `--master local[2]` does not need Pod CRUD via the K8s API; this violates least-privilege.
2. **No securityContext (GAP-2):** All pods have `securityContext: {}`. No `runAsNonRoot`, no `allowPrivilegeEscalation: false`, no capability drops — containers may run as root.
3. **No NetworkPolicy (GAP-3):** Zero NetworkPolicy resources — all pods have unrestricted east-west access across the entire cluster.

**1 WARN (MEDIUM):** `imagePullPolicy: IfNotPresent` allows a tag-mutated cached image to run without re-pulling; digest pinning would be stronger.

**1 INFO (LOW):** Worker replicas reduced to 1 from the standard default of 2, eliminating worker-tier redundancy.
