## Summary

I performed a **read-only** compliance audit of the 3-node RabbitMQ cluster and saved the findings to the `config-audit` ConfigMap (7,691-byte `findings.txt`). No cluster state was modified — every command issued was a `list_*`/`status`/`get` inspection.

**Cluster:** 3/3 nodes healthy (RabbitMQ 3.12.14 / Erlang 25.3.2.15), no alarms, no partitions.

**Key findings flagged for the security team:**

- **HIGH — No TLS:** `ssl_listeners` and `ssl_options` are empty. AMQP is plaintext on 5672 (no 5671), and the management UI/API is plain HTTP on 15672.
- **HIGH — Over-privileged accounts:** `app-user` (on `/app`) and `ops-user` (on `/ops`) both hold full wildcard `.* / .* / .*` configure/write/read permissions — violates least-privilege for service accounts.
- **MEDIUM — Deprecated HA:** the `ha-all` policy on `/app` uses classic mirrored-queue HA (removed in 4.x); both `app-queue` (725 msgs) and `ops-queue` are classic, not quorum. `ops-queue` has no HA policy at all.
- **MEDIUM — Unencrypted metrics/mgmt endpoints** (15692 / 15672).
- **LOW — No user/vhost resource limits** configured.
- **Positive:** default `guest` user removed; queues durable; services ClusterIP-only.

**Config inventory captured:** 3 vhosts (`/`, `/app`, `/ops`), 3 users, per-vhost permissions/policies/queues, plugins, runtime parameters, and listener/TLS settings.

The findings are documented for review only — nothing was remediated, and the live configuration remains exactly as found.
