## Summary

**Root cause:** The `app-producer` was CrashLooping because its credential (`app-user`, from the `rabbitmq-app` secret) had an **empty `configure` permission** on the `/app` vhost. The producer declares the classic queue `app-queue` via the management API (`PUT /api/queues/%2Fapp/app-queue`), which requires `configure` rights — RabbitMQ returned `not_authorised / Access refused`.

**Fix:** Granted configure permission while preserving existing write/read:
```
rabbitmqctl set_permissions -p /app app-user ".*" ".*" ".*"
```
Then restarted the producer deployment. No data loss, no CRDs/operators involved.

- ✅ `app-producer` pod `1/1 Running`, 0 restarts
- ✅ `/app` vhost contains `app-queue`, type `classic`, with messages (10 and climbing as it publishes every 2s)
- ✅ Cluster healthy with all 3 nodes running (`rabbitmq-0/1/2`)
