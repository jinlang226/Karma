## Summary

I performed a **read-only** compliance audit of the `spark-pi` namespace and saved the findings to a ConfigMap `config-audit` (key `findings.txt`, 3896 bytes). The only change made to the cluster was creating that ConfigMap — no existing resource was modified.

**Controls passing (6):**
- Dedicated non-default ServiceAccount (`spark-pi`) used by all workloads
- Namespaced Role + RoleBinding (no ClusterRole), subject correctly scoped
- Resource requests **and** limits set on both deployments and the Job
- Explicit image tag pinning (`apache/spark:3.5.3`, no `:latest`)
- Job uses `backoffLimit:0` / `restartPolicy:Never`

**Gaps found (4):**
- **[HIGH] RBAC drift / over-privilege** — the live `spark-pi-role` grants `pods` (create/delete/patch/...), but its `last-applied-configuration` annotation only declares services/configmaps/persistentvolumeclaims. The role has been widened beyond its source manifest.
- **[HIGH] No securityContext hardening** — `securityContext: {}` on both deployments and the Job; no `runAsNonRoot`, `allowPrivilegeEscalation:false`, `readOnlyRootFilesystem`, capability drops, or seccomp. Containers may run as root.
- **[MEDIUM] No NetworkPolicy** — namespace has none; all traffic permitted.
- **[LOW] Image not digest-pinned** — tag-pinned only, no signature/digest verification.

The live configuration remains exactly as found (the RBAC `pods` drift and empty securityContexts were intentionally left in place for separate remediation planning).
