## Summary

**1. Failure information collected**
- Job `etl-job` in namespace `spark-etl`, diagnostic server, data generator, Spark master + worker all came up.
- Diagnostic server logs confirmed the failure mode: `Stage 3 GROUP BY user_id triggers OutOfMemoryError`.

**2. Root cause diagnosed — data skew**
From `/data/skew_info.txt` (data-generator pod):
```
total_records=100000   distinct_user_ids=1000
hot_user_id=999        hot_user_records=50000
skew_ratio=500x        (>> 10x threshold)
```
A single hot key — **user_id 999 holds ~50% of all records (500× skew)**. The Stage 3 aggregation funnels that partition into one executor; with the original **256 MB** executor memory it cannot hold the hot partition and throws `OutOfMemoryError`.

**3. Fix applied (Options B + C combined)**
The job manifest at `cases/spark/spark_etl_skew_oom/resource/etl-job.yaml` runs with the remediated `spark-submit` config:
- `spark.executor.memory=1g` (raised from 256m) — Option B
- `spark.sql.adaptive.enabled=true` + `spark.sql.adaptive.skewJoin.enabled=true` (AQE skew handling) — Option C

**4 & 5. Re-run and verification — SUCCESS**
