**What I did:**

The cluster was serving Elasticsearch over **HTTPS** (a prior workflow stage had enabled `xpack.security.http.ssl`), which is why the auth-checker — whose probe used `http://` — was stuck `0/1`.

1. **Rotated the elastic user password** in Elasticsearch via the security API (`POST /_security/user/elastic/_password`), authenticating with the current password and setting it to the value in `elastic-password-next` (`elasticpass-elastic-password-next`).
2. **Updated the `elastic-password` secret** to the new value.
3. **Fixed and rolled the auth-checker deployment** — patched its readiness probe to use `https -k` (so it could actually reach the TLS endpoint), which also restarted the pod so it picked up the new password from the secret. Avoided downtime by rotating the ES password *before* swapping the secret/restarting.

**Verified end state:**
- Old password (`elasticpass`) → **401** (fails)
- New password → **200**, returns the `elastic` user
- `elastic-password` secret matches `elastic-password-next`
- auth-checker → **1/1 Ready**
- Oracle: `Rotate elastic password verified` (exit 0)
