## Results

### Diagnosis
The deployment was structurally correct — the Spark master/worker cluster came up cleanly and the baseline job ran successfully using `--master local[2]` (local Spark mode). No configuration fixes were needed beyond what the precondition already applied.

### Performance Comparison

| Strategy | Elapsed | Optimization | Benefit |
|---|---|---|---|
| **Baseline** | 24.59s | None | Skewed shuffle join (10x partition imbalance) |
| **Broadcast Join** | 12.39s | `broadcast_join` | No shuffle, no skew — 2x faster |
| **AQE** | 13.49s | `aqe_skew_join` | Auto-splits hot partitions — 1.8x faster |

### How each optimization helps

- **Broadcast Join**: The small `products` table (100 rows) is broadcast to all executors, eliminating the shuffle entirely. Hot product_id keys (1–5) no longer cause straggler partitions.
- **AQE (Adaptive Query Execution)**: Spark detects skewed partitions at runtime and automatically splits them into smaller sub-tasks, distributing work more evenly.

Both strategies reduced runtime from ~24.6s to ~12–13s (~2x speedup) by avoiding the root cause: a hash-partitioned shuffle where 50% of orders land on 5 product_id keys.
