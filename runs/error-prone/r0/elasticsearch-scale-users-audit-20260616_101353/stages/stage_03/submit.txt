The oracle passes. Here's a summary of what was done:

**Root cause**: The precondition probe found 7 running pods from prior stages, so it skipped the `apply` block that would have set up the file-realm secrets and `es-cluster` StatefulSet. The running ES 8.11.1 cluster had no file realm configured, so `report-user` had no way to authenticate.

**Fix applied**:
1. Deployed `file-realm-gen` pod (ES 7.17.9) to generate bcrypt-hashed credentials for both `ops-user` and `report-user`
2. Created `es-file-realm-aggregate` secret with merged `users` and `users_roles` files containing both users
3. Updated `es-config` ConfigMap to explicitly declare `file1` realm (order 0) and `native1` realm (order 1)
4. Patched the `es` StatefulSet to mount the file-realm secret at `/usr/share/elasticsearch/config/users`, `users_roles`, and `roles.yml`
5. Created a native `read` role in ES 8.x (it's a built-in index privilege in 7.x but not a pre-built role in 8.x)
6. Deleted orphaned `test-index` whose shards were lost when `emptyDir`-backed pods restarted, restoring cluster health to green
