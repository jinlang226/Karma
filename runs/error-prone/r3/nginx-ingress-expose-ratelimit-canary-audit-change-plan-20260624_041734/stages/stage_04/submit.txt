The `config-audit` ConfigMap is in the `demo` namespace with non-empty `findings.txt`. No cluster resources were modified. Here's a summary of the 10 findings documented:

**HIGH (2)**
1. **Rate-limit burst=0** — `rate-api` sets `limit-rps=1` + `limit-burst-multiplier=1`, which computes `burst=(1×1)−1=0` — an invalid nginx value. The controller logged repeated `[emerg] invalid burst value` failures and refused to reload for ~30s. Also returns 503 instead of RFC-standard 429.
2. **No TLS on any Ingress** — all 6 Ingress resources are plain HTTP with no ssl-redirect configured globally or per-resource.

**MEDIUM (4)**
3. **Orphaned canary** — `canary-alt` declares canary for host `canary-alt.example.com` but no primary Ingress exists for that host; logs show continuous "cannot merge alternative backend" errors and the backend is silently dropped.
4. **Inconsistent rate-limit coverage** — `rate-health` (`/health` on `rate.example.com`) has no rate limit while `rate-api` (`/api` on the same host) does; logs confirm `/health` is unthrottled during burst.
5. **Conflicting controller flags** — `--watch-ingress-without-class=true` (from original deploy) and `=false` (patched in revision 2) both appear in the live args list.
6. **No resource limits** — controller has only requests (cpu:100m, mem:90Mi), no limits.

**LOW (3)**
7. `terminationGracePeriodSeconds: 0` — SIGKILLs the pod instantly, defeating the `/wait-shutdown` preStop hook.
8. `readOnlyRootFilesystem: false` — container filesystem is writable.
9. Single replica, no PodDisruptionBudget — any voluntary disruption causes a full ingress outage.

**INFO (1)**
10. Controller ConfigMap is empty — no global security hardening (SSL ciphers, HSTS, log format, etc.).
