All goals achieved:

**What was wrong:** The `es-config` ConfigMap was missing `cluster.initial_master_nodes`. Elasticsearch 7.x requires this setting to bootstrap a brand-new cluster — without it, nodes start but none can initiate a master election, so they all spin indefinitely with "master not discovered yet."

**What was done:**
1. **Added `cluster.initial_master_nodes: [es-cluster-0, es-cluster-1, es-cluster-2]`** to the ConfigMap — this allowed the master election to proceed on restart.
2. **Rolling restart (ordered, pod-by-pod)** — each pod restarted and came back ready before the next was touched, keeping the cluster available.
3. **Cluster formed** (green, 3 nodes, master elected).
4. **Removed `cluster.initial_master_nodes`** from the ConfigMap — this setting must not persist; if it remained, a future accidental re-bootstrap could split the cluster state.
5. **Second rolling restart** — confirmed the cluster stays green and a master is re-elected without the bootstrap setting. The cluster state is now persisted on each node's data volume, so discovery works via `discovery.seed_hosts` alone on all future restarts.

**Final state:** cluster `green`, 3/3 nodes ready, master on es-cluster-1, `cluster.initial_master_nodes` absent from both the ConfigMap and running node configuration.
