Done. Here's what was done and why it works:

**Problem:** With only 2 voting members, `majorityVoteCount` was 2 — meaning both data nodes had to be up for a primary to exist. Losing either caused an election stalemate.

**Fix:** Added `mongo-arb-0` as an arbiter via `rs.addArb()`.
- `setDefaultRWConcern` to `w:1` was required first — MongoDB 5+ blocks reconfigs that change the implicit default write concern without an explicit cluster-wide setting.

**Result:**
- `votingMembersCount: 3`, `majorityVoteCount: 2`
- Elections now need 2 of 3 votes — the surviving data node + arbiter can elect a primary if the other data node goes down
- No extra storage: the arbiter holds no data and its pod was already running
- App database `app.test` still has all 3 documents intact
