All checks pass. Here's what was done and why:

**Root cause**: The ES cluster (deployed in stage_03 with ES 8.11.1) had no file realm configured. Two secrets existed — `es-file-realm-aggregate` with only `ops-user`, and `user-provided-file-realm` with only `report-user` — but neither was mounted into the pods nor was the file realm activated in `elasticsearch.yml`.

**Fix (3 changes)**:

1. **Merged `es-file-realm-aggregate`** — concatenated both secrets so the aggregate now holds both `ops-user` (superuser) and `report-user` (viewer) with their bcrypt hashes and role assignments.

2. **Updated `es-config` ConfigMap** — added `xpack.security.authc.realms.file.file1.order: 0` and `xpack.security.authc.realms.native.native1.order: 1` to `elasticsearch.yml`, keeping all existing TLS cert paths intact.

3. **Patched the StatefulSet** — added a `file-realm` volume from `es-file-realm-aggregate` and mounted `users` and `users_roles` at `/usr/share/elasticsearch/config/`, triggering a rolling restart that completed with all 3 nodes healthy (green cluster).

Also seeded the `app-data` index (the precondition seed loop had previously failed silently because file realm wasn't active yet when it ran).
