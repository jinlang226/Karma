ROOT CAUSE:
Stage 3 (aggregation by user_id) fails with OutOfMemoryError because:
1. The Spark job was configured with only 256m driver/executor memory — below
   Spark's ~471MB reserved system memory minimum — causing the JVM to abort
   with "must be at least X, please increase heap size."
2. Data is severely skewed: user_id=999 holds 50,000 of 100,000 records
   (skew_ratio=500x >> 10x threshold). The groupBy(user_id) in Stage 3
   collapses all 50k records onto one partition, exhausting the 256m heap.

DIAGNOSIS:
- kubectl get job etl-job -n spark-etl → STATUS: Failed (OOM in Stage 3)
- kubectl logs -n spark-etl -l app=etl-job → OutOfMemoryError / "must be at
  least ... increase heap size" in driver log
- kubectl exec data-generator -- cat /data/skew_info.txt:
    hot_user_id=999, hot_user_records=50000/100000, skew_ratio=500x
- diagnostic-server: "Stage 3 GROUP BY user_id triggers OutOfMemoryError"

FIX APPLIED (cases/spark/spark_etl_skew_oom/resource/etl-job.yaml):
- Raised --driver-memory from 256m → 1g
- Raised spark.executor.memory from 256m → 1g  (executor runs in driver JVM
  under local[2] mode, so both must be raised)
- Added --conf spark.sql.adaptive.enabled=true
- Added --conf spark.sql.adaptive.skewJoin.enabled=true  (splits the
  user_id=999 skewed partition automatically)

VERIFICATION:
