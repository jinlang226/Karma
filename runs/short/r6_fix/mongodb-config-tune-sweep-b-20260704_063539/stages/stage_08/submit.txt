The replica set is stable: 1 PRIMARY (`mongodb-replica-2`) and 2 SECONDARY (`mongodb-replica-0`, `mongodb-replica-1`), all pods `1/1 Ready`.

**What was wrong and what was fixed:**

The probe configuration had two critical flaws that caused cascading pod restarts and repeated elections (the replica set was on election term 16):

1. **`timeoutSeconds: 1`** — `mongosh` startup overhead alone takes ~0.66s under normal conditions; any load spike caused timeouts and false-negative failures.

2. **`failureThreshold: 2` on the liveness probe** — only 2 consecutive failures (40s total) before Kubernetes kills the pod. MongoDB elections and startup bursts easily exceeded this.

**Fixed values applied via `kubectl patch`:**

| Parameter | Before | After |
|---|---|---|
| Readiness `timeoutSeconds` | 1s | 5s |
| Readiness `failureThreshold` | 2 | 6 |
| Readiness `initialDelaySeconds` | 5s | 10s |
| Liveness `timeoutSeconds` | 1s | 5s |
| Liveness `failureThreshold` | 2 | 3 |
| Liveness `initialDelaySeconds` | 15s | 30s |

The probe command itself (`db.adminCommand('ping').ok`) is correct — `ping` is one of the few commands MongoDB allows without authentication even when `authorization: enabled`.
