The Metricbeat index grew from 10 to 290 docs, confirming continuous data flow. Everything is working correctly.

Here's a summary of what was done:

**Problem**: Elasticsearch pods had no monitoring sidecars, so no metrics or logs were being shipped to the monitoring cluster.

**Solution**: Added two sidecar containers to every Elasticsearch pod via a StatefulSet patch:

1. **Metricbeat sidecar** (`metricbeat:7.17.9`): Connects to the local ES node at `https://localhost:9200` using the `elasticsearch` module with `xpack.enabled: true`, which writes Stack Monitoring format metrics to `.monitoring-es-7-mb-*` indices in the monitoring cluster.

2. **Filebeat sidecar** (`filebeat:7.17.9`): Reads ES server logs from a shared `es-logs` emptyDir volume and ships them to `.monitoring-es-7-filebeat-*` indices in the monitoring cluster.

**Supporting changes**:
- `es-log4j2-with-file` ConfigMap: Extended the ES log4j2 config with a `RollingFile` appender (alongside the existing Console appender) that writes ECS-format JSON logs to the shared `es-logs` volume — making logs readable by the Filebeat sidecar.
- Loaded `metricbeat:7.17.9` and `filebeat:7.17.9` images into the kind cluster.

**Result**:
- All 3 ES pods: `3/3 Running` with `elasticsearch`, `metricbeat`, `filebeat` containers
- ES cluster: green, 3 nodes
- Monitoring cluster: `.monitoring-es-7-mb-2026.07.03` (metrics, 290+ docs) and `.monitoring-es-7-filebeat-2026.07.03` (logs, 92+ docs) — both actively receiving data
