All goals are achieved. Here's a summary of what was done and why:

**Root cause:** The `es-config` ConfigMap was missing `cluster.initial_master_nodes`, which Elasticsearch 7.x requires for the very first cluster bootstrap. Without it, nodes can't form a quorum to elect a master.

**Fix applied:**

1. **Added `cluster.initial_master_nodes`** to the ConfigMap — lists all 3 nodes so any quorum of them can bootstrap the cluster.

2. **Restarted all 3 pods** — required because the config is mounted as a `subPath` volume, which doesn't auto-refresh when the ConfigMap changes.

3. **Cluster formed**: green, 3 nodes, master elected (`es-cluster-2` → later `es-cluster-1` after rolling restart).

4. **Removed `cluster.initial_master_nodes`** from the ConfigMap — this setting is bootstrap-only. Leaving it in place after the cluster forms is dangerous: on a future full restart it could trigger a new bootstrap election and ignore the stored cluster state, risking split-brain.

5. **Rolling restart** (one pod at a time, waiting for green+3 nodes before proceeding): cluster remained green throughout and all three nodes restarted cleanly without the bootstrap setting.

**Final state:** green, 3 nodes, master elected (`es-cluster-1`), `cluster.initial_master_nodes` absent from config.
