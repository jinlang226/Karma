Everything is working. Here's what was done:

**Problem:** The `prometheus-config` ConfigMap only had a self-scrape job (`localhost:9090`) with no RabbitMQ targets.

**Fix:** Updated the ConfigMap with two additions, then restarted the Prometheus deployment to pick up the changes:

1. **Scrape job** targeting each RabbitMQ pod individually via the headless service DNS names (`rabbitmq-{0,1,2}.rabbitmq-headless.<ns>.svc.cluster.local:15692`). The headless service already existed and exposes port 15692, so no new Service was needed.

2. **Recording rule** `rabbitmq_up = up{job="rabbitmq"}` — the built-in RabbitMQ Prometheus plugin (port 15692) doesn't emit a `rabbitmq_up` metric, so a recording rule aliases the standard Prometheus `up` scrape-success metric under that name.

**Result:**
- 3 RabbitMQ pods remain healthy (1/1 Running)
- All 3 Prometheus targets show `health: up`
- `rabbitmq_up` query returns 3 samples, all `= 1`
