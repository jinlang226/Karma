**Problem:** The `prometheus-config` ConfigMap only had a scrape job for `localhost:9090` (Prometheus itself) — no RabbitMQ targets.

**Fix (two patches to `prometheus-config`, then a rollout restart):**

1. **Added a `rabbitmq` scrape job** with static targets pointing to each pod via the existing headless service DNS (`rabbitmq-{0,1,2}.rabbitmq-headless.<namespace>.svc.cluster.local:15692`). The headless service already exposed port 15692 and each pod's `/metrics` endpoint was reachable.

2. **Added a recording rule** (`rules.yml` key in the same ConfigMap, auto-mounted at `/etc/prometheus/rules.yml`) that maps `up{job="rabbitmq"}` → `rabbitmq_up`, since RabbitMQ's native Prometheus plugin doesn't expose a `rabbitmq_up` metric.
