## Upgrade Summary

**RabbitMQ 3.12.14 → 4.1.8** — completed via a staged rolling upgrade using only native Kubernetes resources.

### What was done

1. **Enabled all feature flags** on 3.12 (prerequisite for major version upgrades)

2. **Converted `app-queue` to quorum type** before any rolling restart, so messages would survive pod restarts with emptyDir storage. Drained and saved 551 messages externally as a backup.

3. **Updated ConfigMap** — replaced deprecated `queue_master_locator = min-masters` with `queue_leader_locator = client-local` for RabbitMQ 4.x compatibility.

4. **Staged rolling upgrade** (the only supported upgrade path):
   - 3.12 → 3.13 via `kubectl set image statefulset/rabbitmq`
   - 3.13 → 4.0 with feature flags enabled between steps
   - 4.0 → 4.1

   After each rolling update, one node formed a standalone cluster (emptyDir means fresh state on restart). Fixed each time with `rabbitmqctl stop_app/reset/join_cluster/start_app`.

5. **Restored cluster state** after final upgrade: vhosts (`/app`, `/ops`), users (`app-user`, `ops-user`), permissions, queues, and republished 1,700+ messages.
