Everything is working. Here's what was done:

**Problem:** The `prometheus-config` ConfigMap only had a `static_configs` scrape job targeting `localhost:9090` (Prometheus itself) — no RabbitMQ targets at all.

**Fix:** Updated the ConfigMap with two additions:

1. **RabbitMQ scrape job** — static targets pointing to each pod via the headless service DNS (`rabbitmq-{0,1,2}.rabbitmq-headless.<namespace>.svc.cluster.local:15692`). The headless service gives stable per-pod DNS, so each pod is scraped individually.

2. **Recording rule** (`rabbitmq_up = up{job="rabbitmq"}`) — the built-in RabbitMQ Prometheus plugin doesn't export a `rabbitmq_up` metric; Prometheus generates the synthetic `up` metric per scrape target. The recording rule aliases it to `rabbitmq_up`.

**Result:**
- All 3 RabbitMQ targets show `health: up` in `/api/v1/targets`
- `rabbitmq_up` query returns 3 samples, all value `1`
- All 3 RabbitMQ pods remain healthy (`1/1 Running`, 0 restarts)
