**Problem:** The replica set `rs0` had only 2 voting members (`mongo-rs-0` PRIMARY, `mongo-rs-1` SECONDARY). With an even number of voters, losing either member makes majority (2/2) unachievable and elections stall permanently.

**Fix:** Added the already-running `mongo-arb-0` pod as an **arbiter** member via `rs.addArb()`.

- Had to first run `setDefaultRWConcern` (`w: 1`) — MongoDB 5+ blocks a reconfig that would implicitly change the write concern without an explicit cluster-wide setting.
- The arbiter uses `emptyDir` (no PVC), so **zero storage cost** is added.

- `mongo-rs-0` — PRIMARY (votes: 1, data-bearing)
- `mongo-rs-1` — SECONDARY (votes: 1, data-bearing)
- `mongo-arb-0` — ARBITER (votes: 1, no data)
- `votingMembersCount: 3`, `majorityVoteCount: 2` — elections now succeed even if one member is unavailable
