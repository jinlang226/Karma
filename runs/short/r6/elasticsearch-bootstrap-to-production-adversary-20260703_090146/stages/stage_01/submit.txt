**What was wrong:** The `es-config` ConfigMap was missing `cluster.initial_master_nodes`, which Elasticsearch 7.x requires to bootstrap the first master election. Without it, nodes loop indefinitely with "master not discovered yet".

**What was done:**
1. **Added `cluster.initial_master_nodes`** to the ConfigMap listing all 3 nodes, then restarted all pods simultaneously (safe since none were ready) → cluster formed, elected `es-cluster-1` as master, status **green**.
2. **Removed `cluster.initial_master_nodes`** from the ConfigMap — it is a bootstrap-only setting that must not persist, as its presence after bootstrap can cause split-brain if data is ever wiped on a future restart.
3. **Rolling restart** — deleted pods one at a time, waiting for each to rejoin before proceeding. Cluster stayed **green** throughout with all 3 nodes, confirming the fix is safe for future restarts.

**Final state:**
- Cluster status: `green`, 3 nodes, master elected (`es-cluster-0`)
- Config on disk: no `cluster.initial_master_nodes` — rejoining nodes use `discovery.seed_hosts` + on-disk cluster UUID, which is the correct post-bootstrap behavior
