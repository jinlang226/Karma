## Upgrade Summary: RabbitMQ 3.12 → 4.1

**Result: All requirements met.**

| Requirement | Status |
|-------------|--------|
| All 3 nodes on RabbitMQ 4.1 | ✅ 4.1.8 on Erlang 27.3.4.13 |
| /app vhost exists | ✅ |
| app-queue exists with messages | ✅ 760 messages (growing) |
| Cluster healthy, 3 nodes | ✅ No alarms, no partitions |

**What happened and why:**

- A direct rolling upgrade from 3.12 → 4.1 fails because RabbitMQ 4.x introduces mandatory feature flags unknown to 3.12 — new pods refused to join the existing cluster (`incompatible_feature_flags`).

- The solution was a **blue-green upgrade**:
  1. Exported all 737 queue messages and cluster definitions while 3.12 was healthy
  2. Scaled the StatefulSet to 0 (all nodes down simultaneously)
  3. Set image to `rabbitmq:4.1-management` and scaled back to 3 — all nodes came up on the same version, forming a fresh 4.1 cluster with no incompatibility check
  4. Restored users, vhosts, and permissions via the import API
  5. Recreated app-queue as a classic durable queue (RabbitMQ 4.0 removed the classic HA mirroring policy that the old setup used)
  6. Republished all 737 original messages
