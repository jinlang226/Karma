The cluster is fully restored. Here's a summary of what was done and why:

**Root cause:** The `rabbitmq-cookie-perpod` secret had `driftedcookie` for `rabbitmq-2` while the other nodes had `supersecretcookie`. This was injected by the adversary during setup. Had `rabbitmq-2` been restarted under maintenance, the init container (in the proper failover StatefulSet design) would have copied `driftedcookie` to `.erlang.cookie`, causing `rabbitmq-2` to fail to rejoin the cluster (Erlang authentication mismatch).

**Fixes applied:**
1. **Cookie secret patched** — updated `rabbitmq-cookie-perpod/rabbitmq-2` from `driftedcookie` to `supersecretcookie`, making all 3 pod entries consistent.

2. **StatefulSet updated to the failover design** — replaced the classic_queue STS (which used a shared `RABBITMQ_ERLANG_COOKIE` env var) with the proper failover STS (which uses a `copy-cookie` init container to inject per-pod cookies from the secret). This ensures the cookie mechanism is deterministic and auditable per pod.

3. **Rolling restart completed** — all 3 pods cycled through the new template:
   - `rabbitmq-2` → first (manually triggered), joined the running cluster with the correct cookie
   - `rabbitmq-1` → restarted via STS rolling update, synced cluster data
   - `rabbitmq-0` → restarted last, synced from the two running nodes

**Final state:**
- All 3 pods: `1/1 Running`
- All 3 nodes: in the same cluster as both Disk Nodes and Running Nodes
- No alarms, no network partitions
- Cookie secret: consistent (`supersecretcookie` for all 3)
- Actual `.erlang.cookie` files on all pods: `supersecretcookie` (written by init container)
