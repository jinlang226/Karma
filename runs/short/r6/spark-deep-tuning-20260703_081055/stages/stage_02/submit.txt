## Diagnosis

Root cause: Stage 3 (GROUP BY user_id) triggers OutOfMemoryError because:
1. The ETL job was configured with only 256m of driver/executor memory — below
   Spark's minimum (~471 MB reserved system memory), so the JVM heap is
   exhausted before any real work starts.
2. Severe data skew: user_id 999 holds 50,000 / 100,000 records (50%, skew
   ratio 500x — far above the 10x threshold). In local[2] mode the skewed
   partition cannot be split across executors, concentrating all 50k rows into
   one task within the driver JVM.

## Fix Applied

1. Raised executor/driver memory from 256m → 1g in etl-job.yaml:
     --driver-memory 1g
     --conf spark.executor.memory=1g
   This brings the heap above Spark's minimum and gives the aggregation
   sufficient space.

2. Enabled Adaptive Query Execution (AQE) with skew-join handling:
     --conf spark.sql.adaptive.enabled=true
     --conf spark.sql.adaptive.skewJoin.enabled=true
   AQE detects the hot partition (user_id 999) at runtime and splits it into
   smaller sub-tasks, distributing the load.

3. Scaled spark-worker deployment to 4 replicas with SPARK_WORKER_MEMORY=2G
   for additional parallelism headroom.

## Verification

- kubectl get job etl-job -n spark-etl → Complete (1/1), duration 99s
- Job logs contain: spark_etl_pipeline_verified
- ResourceProfile shows executor memory=1024 MB (1g applied)
- Spark workers: 4 replicas, SPARK_WORKER_MEMORY=2G
- All three stages completed:
    Stage 1: load raw data
    Stage 2: clean and transform
    Stage 3: aggregate by user_id (skewed on user_id 999)
