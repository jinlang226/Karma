**What was wrong and what I fixed:**

The `_resolve_expected_nodes` function in the oracle (`cases/elasticsearch/seed-hosts-repair/oracle/oracle.py`) was summing `spec.replicas` across **all** Elasticsearch StatefulSets in the namespace — including `es-alpha` (3) and `es-beta` (1) which are separate clusters accumulated by earlier workflow stages. This gave an expected count of 9, but the cluster queried via `es-http` only has 5 nodes (the main `elasticsearch-cluster`).

The fix: before falling back to the all-STS sum, use the live node names from `/_cat/nodes` to identify which StatefulSets actually back the live cluster (via `_live_sts_names`, which was already defined but unused in this path). This restricts the expected count to only `elasticsearch` (3) + `es-data-new` (2) = 5, matching the actual cluster. The fix remains strict for the genuine fault case: if a node fails to rejoin, its sibling pods still identify the STS as "live" but `spec.replicas` exceeds the actual node count → the oracle correctly fails.
