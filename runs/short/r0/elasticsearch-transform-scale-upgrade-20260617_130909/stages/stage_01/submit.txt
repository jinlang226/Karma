**Resources created in `elasticsearch` namespace:**

| Resource | Name | Purpose |
|---|---|---|
| ConfigMap | `es-config` | `elasticsearch.yml` with TLS, discovery, and security settings |
| Service (headless) | `es-internal` | Pod-to-pod transport discovery (`publishNotReadyAddresses: true`) |
| Service (ClusterIP) | `es-http` | Client access on port 9200 |
| PodDisruptionBudget | `es-pdb` | `maxUnavailable: 1` |
| StatefulSet | `es` | 3 replicas, `podManagementPolicy: Parallel` |
| PVCs | `data-es-{0,1,2}` | 10Gi each via `standard` storage class |

**Key design decisions:**
- `podManagementPolicy: Parallel` — all 3 pods start simultaneously so they can form a quorum; sequential startup would deadlock (es-0 needs peers to elect a master before becoming Ready)
- `publishNotReadyAddresses: true` on the headless service — lets nodes discover each other before readiness probes pass
- Same `es-http-tls` cert used for both HTTP and transport TLS with `verification_mode: certificate` (skips hostname check, validates cert chain only)
- `ELASTIC_PASSWORD` injected from the existing `elastic-password` secret
