All pods are ready and the replica set is healthy: **1 PRIMARY** (`mongodb-replica-1`) and **2 SECONDARYs** (`mongodb-replica-0`, `mongodb-replica-2`).

**Root cause:** The `health-overrides` ConfigMap contained a stale entry `mongodb-replica-1: legacyHealthPass123`. The health-check script checks this ConfigMap for per-pod password overrides before falling back to the shared secret — so `mongodb-replica-1` was authenticating with the wrong password, causing the readiness probe to fail with `Authentication failed`.

**Fix:** Removed the stale `mongodb-replica-1` key from the `health-overrides` ConfigMap. Once Kubernetes refreshed the mounted volume (~30s), the pod's health-check started using the correct password from the shared secret (`healthpass123`) and the readiness probe began succeeding.
