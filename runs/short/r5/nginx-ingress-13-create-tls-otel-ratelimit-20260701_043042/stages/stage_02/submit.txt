The fix is complete. Here's what was wrong and what was done:

**Root cause:** The `expired-tls-secret` Kubernetes TLS secret contained a certificate valid only from Jan 1–2, 2020 — deliberately expired. The `demo-ingress` resource was configured to use this secret for TLS termination of `demo.example.com`, causing all HTTPS connections to fail certificate validation.

**Fix:**
1. Identified the CA key at `/tmp/test-ca.key` (the same CA whose cert is mounted into the `curl-test` pod at `/tmp/tls/ca.crt` via the `test-ca` ConfigMap)
2. Generated a new 2048-bit RSA key and leaf certificate for `demo.example.com` with SAN, valid until June 2028, signed by that CA
3. Replaced the secret: `kubectl apply` with the new cert/key over the existing `expired-tls-secret`
4. Fixed the empty `INGRESS_NODE_IP` in `/tmp/ingress_env` (set to `172.18.0.6`, the node's internal IP)

curl now returns `hello` with exit code 0 and `SSL certificate verify ok`.
