Both goals achieved:

1. **Insecure access rejected** — `--insecure` returns `ERROR: node is running secure mode, SSL connection required` (exit 1)
2. **Secure access succeeds** — `--certs-dir=/cockroach/cockroach-certs` returns `1` (exit 0)

**What was done:**
1. Used `cockroach cert` inside `crdb-cluster-0` to generate a CA cert, a shared node cert (with all 3 node DNS names as SANs), and a root client cert
2. Created two Kubernetes secrets: `cockroachdb-node` (ca.crt + node.crt + node.key) and `cockroachdb-client-root` (ca.crt + client root certs)
3. Patched the StatefulSet to replace `--insecure` with `--certs-dir=/cockroach/cockroach-certs`, mount the certs secret via a projected volume at that path, and switch the liveness/readiness probes from HTTP to HTTPS
4. Scaled to 0 then back to 3 to restart all nodes simultaneously (required because CockroachDB can't operate in mixed insecure/secure mode during a rolling update)
