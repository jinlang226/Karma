All goals are met:

- **app-producer** is `1/1 Running` (Ready), no restarts since the fix.
- **`/app` vhost** contains **`app-queue`** of type **`classic`**, and messages are accumulating (10 and growing).
- **Cluster** remains healthy with all **3 nodes** (`rabbitmq-0/1/2`) running as disk nodes.

**Root cause:** the setup job granted `app-user` write/read permissions on `/app` but left `configure` permission empty (`""`), so the management API rejected the `PUT /queues/%2Fapp/app-queue` declare call with `not_authorised`, crash-looping the producer. Fixed by running `rabbitmqctl set_permissions -p /app app-user ".*" ".*" ".*"` to grant configure rights — no data loss, no CRDs/operators involved. (Note: the job's `force-quorum` policy also silently failed — `queue-type` isn't a valid RabbitMQ policy key — so it never actually threatened the classic queue type; no action was needed there.)
