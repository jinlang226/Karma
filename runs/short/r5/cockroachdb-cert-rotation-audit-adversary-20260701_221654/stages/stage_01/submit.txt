All three verifications pass:

1. **Secure SQL works** — `SELECT 1` returns `1` via TLS on `crdb-cluster-0`.
2. **Node cert rotated** — fingerprint `5D:0D:C0:63:...` confirms the new cert is in the secret.
3. **Cert validity ~1 year** — `notAfter=Jul 1 22:19:21 2027 GMT` (expires in ~365 days).

**What was done:**
- The adversary had deleted the `crdb-cluster-certs` secret; pods were still running on their in-memory mount.
- Generated a new 2048-bit node key + CSR (using the existing `node.cnf` with all required SANs) in the `openssl-toolbox` pod.
- Signed the new node cert with the CA key for 365 days; same for `client.root` cert.
- Recreated the `crdb-cluster-certs` secret with `ca.crt`, `node.crt`, `node.key`, `client.root.crt`, `client.root.key`.
- Performed a rolling restart of the `crdb-cluster` statefulset; all 3 nodes came back healthy.
