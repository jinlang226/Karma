Both goals are met:

1. **Insecure access rejected** — `cockroach sql --insecure` returns `SSL connection required` (exit code 1).
2. **Secure access succeeds** — `cockroach sql --certs-dir=/cockroach/cockroach-certs` returns `1` (exit code 0).

**What was done:**

- Generated a CA certificate, node certificate (with all DNS SANs for the 3 nodes, services, and localhost), and a root client certificate using `cockroach cert` commands inside an existing pod.
- Created a single `crdb-certs` Kubernetes Secret containing all five cert files (`ca.crt`, `node.crt`, `node.key`, `client.root.crt`, `client.root.key`).
- Updated the StatefulSet to remove `--insecure`, add `--certs-dir=/cockroach/cockroach-certs`, mount the secret at that path, and switch liveness/readiness probes from HTTP to HTTPS.
- Forced a simultaneous restart of all three pods (a rolling update would have left a mixed secure/insecure cluster that couldn't form quorum).
