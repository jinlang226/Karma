Everything checks out. Summary of the fix:

- **Root cause**: `prometheus-config` ConfigMap only had a `localhost:9090` self-scrape job — nothing pointed at RabbitMQ. RabbitMQ itself was already correctly configured (plugin enabled, port 15692 exposed on the `rabbitmq`/`rabbitmq-headless` Services).
- **Fix**: Patched `prometheus-config` (native ConfigMap, no CRDs) to add a `rabbitmq` scrape job with static targets for the 3 stable per-pod DNS names via the headless Service (`rabbitmq-{0,1,2}.rabbitmq-headless.<ns>.svc.cluster.local:15692`), plus a `rules.yml` recording rule (`rabbitmq_up = up{job="rabbitmq"}`) since the RabbitMQ Prometheus plugin doesn't itself expose a metric literally named `rabbitmq_up`.
- **Applied via** `kubectl apply` + `/-/reload` (enabled via `--web.enable-lifecycle`) — no pod restarts needed.
- **Verified**: all 3 RabbitMQ targets show `health: up`, `rabbitmq_up` query returns 3 samples (all value `1`), and `rabbitmqctl cluster_status` confirms the 3-node cluster is still fully healthy.
