The 3-node CockroachDB cluster is fully up. Here's a summary of what was deployed and the one non-obvious fix required:

**Resources created:**
- `Service/crdb-cluster` — headless service (grpc 26257, http 8080), `publishNotReadyAddresses: true`
- `Service/crdb-cluster-public` — standard ClusterIP service (same ports)
- `StatefulSet/crdb-cluster` — 3 replicas, `cockroachdb/cockroach:v24.1.0`, insecure mode, 10Gi PVC each
- `PodDisruptionBudget/crdb-cluster-pdb` — `minAvailable: 2` to preserve quorum during voluntary disruptions

**All 3 nodes reporting `is_available: true, is_live: true`** with their full FQDN addresses.

**The key issue encountered and fixed:** CockroachDB uses mutual gossip — node 0 must connect back to node 1 using node 1's advertised hostname. With a standard headless service, DNS entries are only created for **Ready** pods, so `crdb-cluster-1.crdb-cluster...` didn't resolve until node 1 was already ready — a deadlock. Setting `publishNotReadyAddresses: true` on the headless service tells CoreDNS to register all Running pods (not just Ready ones), which broke the cycle.
