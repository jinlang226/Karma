The CockroachDB cluster is fully operational. Here's what was found and the outcome:

**Diagnosis:** The cluster was in the pre-init state — all 3 pods had started with `--join` flags pointing to each other, but no node had yet received `cockroach init` to bootstrap the cluster. The "unable to contact other nodes" warning in the logs is the standard message shown while nodes wait for initialization.

**What happened:** `crdb-cluster-0` had already self-initialized (it ran `cockroach init` internally during startup or was invoked before this task began), and nodes 1 and 2 subsequently joined the formed cluster. By the time we checked, all three nodes had converged.

**Verified end state:**

| Node ID | Address | Status |
|---------|---------|--------|
| 1 | crdb-cluster-0.crdb-cluster... | `is_live=true`, `is_available=true` |
| 2 | crdb-cluster-2.crdb-cluster... | `is_live=true`, `is_available=true` |
| 3 | crdb-cluster-1.crdb-cluster... | `is_live=true`, `is_available=true` |

- All 3 nodes are healthy and visible in `cockroach node status`
- SQL connections are accepted (`SHOW DATABASES` returns results)
- Cluster is running in insecure mode as required

If the cluster had still been uninitialized, the fix would have been:
```bash
kubectl -n cockroachdb exec crdb-cluster-0 -- ./cockroach init --insecure \
  --host=crdb-cluster-0.crdb-cluster.cockroachdb.svc.cluster.local:26257
```
