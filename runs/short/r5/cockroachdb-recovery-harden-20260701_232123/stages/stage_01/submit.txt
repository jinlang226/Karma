All three verification checks pass:

1. **All pods Ready** — `crdb-cluster-0`, `crdb-cluster-1`, `crdb-cluster-2` all show `1/1 Running`
2. **All nodes live** — `is_available=true`, `is_live=true` for all 3 nodes
3. **SQL works** — `SELECT 1` succeeds

**Root cause:** The StatefulSet command had a deliberate fault injected — a conditional that set `HTTP_ADDR="127.0.0.1:8080"` specifically when `POD_NAME = "crdb-cluster-1"`. Since Kubernetes readiness probes are issued by the kubelet from outside the pod, they couldn't reach the HTTP endpoint bound only to localhost. The pod was permanently stuck in Not Ready, preventing cluster recovery.

**Fix:** Patched the StatefulSet to unconditionally use `HTTP_ADDR="0.0.0.0:8080"`, removing the bad conditional. The rolling update cycled all three pods and the cluster is now fully healthy.
