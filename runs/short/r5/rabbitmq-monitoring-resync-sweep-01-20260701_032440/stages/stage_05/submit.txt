All requirements are met:

- **3 RabbitMQ targets UP** in Prometheus (`rabbitmq-0/1/2` on port 15692, `/metrics`).
- **`rabbitmq_up` query returns 3 samples**, all value `1`.
- **RabbitMQ cluster healthy** with all 3 nodes running and clustered.

Root cause: two pieces of already-authored config weren't wired together — Prometheus's `scrape_configs` never listed the 3 RabbitMQ pod targets, and a recording rule ConfigMap (`prometheus-rules`, defining `rabbitmq_up = up{job="rabbitmq"}`) was mounted into the pod but never referenced via `rule_files` in `prometheus.yml`. Fixed by updating the `prometheus-config` ConfigMap to add:
1. A `rabbitmq` scrape job with static targets for each pod via the `rabbitmq-headless` service DNS names.
2. A `rule_files: [/etc/prometheus/rules/*.yml]` entry to load the existing recording rule.

Then reloaded via the `/-/reload` endpoint (no pod restarts needed, RabbitMQ was never touched).
