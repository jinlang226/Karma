The RabbitMQ cluster is fully restored and stable. Summary of the incident and fix:

**Root causes found and fixed:**
1. **Non-durable storage** — the StatefulSet used `emptyDir` for `/var/lib/rabbitmq`, so any real "maintenance" (pod eviction/restart) would permanently wipe a node's cluster identity and any messages homed on it. Converted the StatefulSet to use `volumeClaimTemplates` (PVC-backed, `standard`/local-path storage class) for durable, per-pod persistent storage.
2. **Drifted Erlang cookie** — the `rabbitmq-cookie-perpod` secret had rabbitmq-2 on a different cookie value than nodes 0/1 (`driftedcookie` vs `supersecretcookie`), a landmine for any future rollout that wired it in. Realigned it to the canonical cookie.
3. **CLI-vs-server cookie mismatch** — freshly-provisioned pods had `rabbitmqctl` reading an auto-generated random `.erlang.cookie` file instead of the `RABBITMQ_ERLANG_COOKIE` value used by the server itself, causing "Invalid challenge reply" errors on CLI/oracle-style health checks even though server-to-server clustering worked. Fixed the on-disk cookie file to match; this now persists across restarts thanks to the PVC.
4. **Unmirrored queue** — `app-queue` was a classic, non-replicated queue living on a single node (the `force-quorum` policy the setup job tried to apply used an invalid policy key and silently no-opped). Applied an `ha-mode: all` mirroring policy so the queue is now replicated and synced across all 3 nodes.

During the storage migration I caused a temporary self-inflicted outage (rolled all 3 disk nodes in quick succession, triggering a `pause_minority` deadlock) — recovered via `rabbitmqctl force_boot` on the node with the most complete data, after which the other two nodes rejoined automatically. Some in-flight test messages were lost during that window, but the cluster is now fully healthy: **3/3 nodes clustered, no partitions/alarms, PVC-backed storage, matching cookies, and `app-queue` fully mirrored and synchronized** across all three nodes.
