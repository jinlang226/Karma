Oracle passes. Here's what was done:

**Root cause:** The Elasticsearch cluster had no file realm configured — the `es-file-realm-aggregate` secret (ops-user only) and `user-provided-file-realm` secret (report-user) existed but were neither merged nor mounted into the pods.

**Changes made:**

1. **Merged secrets** — Updated `es-file-realm-aggregate` to include both `ops-user` and `report-user` entries (users + users_roles from the `user-provided-file-realm` secret).

2. **Enabled file realm** — Added `xpack.security.authc.realms.file.file1.order: 0` to the `es-config` ConfigMap's `elasticsearch.yml`.

3. **Mounted realm files** — Patched the `es-cluster` StatefulSet to add the `es-file-realm-aggregate` secret as a volume, with `subPath` mounts placing `users` and `users_roles` at `/usr/share/elasticsearch/config/`.

4. **Rolling restart** — The StatefulSet performed an automatic rolling restart (all 3 pods updated, cluster stayed healthy/green throughout).

5. **Seeded index** — Created the `app-data` index and a seed document (the precondition seeding had failed because the file realm wasn't configured at the time it ran).

`report-user` can now read `app-data` (viewer role grants index read), `ops-user` retains superuser access, and the `elastic` built-in user is unaffected.
