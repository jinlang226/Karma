Everything is complete and verified. Here's a summary of what was done and why:

**Root cause:** The `es-config` ConfigMap was missing `cluster.initial_master_nodes`. In Elasticsearch 7.x, this setting is mandatory on first boot to allow nodes to agree on cluster quorum and elect a master. Without it, nodes keep searching for a master that never forms.

**Fix applied:**
1. **Added `cluster.initial_master_nodes`** to the ConfigMap listing all 3 node names — this gives ES the quorum information needed to bootstrap.
2. **Restarted all pods simultaneously** (the ConfigMap is mounted via `subPath`, so file changes only take effect on pod restart). All 3 came up together and elected `es-cluster-2` as master.
3. **Removed `cluster.initial_master_nodes`** from the ConfigMap immediately after the cluster formed. This setting is bootstrap-only — leaving it active risks accidental re-bootstrapping if cluster state is ever lost.
4. **Rolling restart** via `kubectl rollout restart` to propagate the clean config to all running nodes. The cluster stayed **green throughout** (all 3 nodes remained available, shards were never unassigned).

**Final state:** Cluster is green, 3/3 nodes, master elected (`es-cluster-2`), 6 active shards, 0 pending tasks. The persisted configmap has only `discovery.seed_hosts` (for ongoing peer discovery) and no bootstrap-only settings — safe for any future restart.
