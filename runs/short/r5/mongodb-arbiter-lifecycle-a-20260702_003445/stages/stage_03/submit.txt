Elections are now stable. Here's what was done:

**Root cause:** The arbiter pod (`mongo-arb-0`) was running but never added to the replica set. With only 2 voting members, losing either one meant no majority could form (required 2 of 2 votes), causing election failure.

**Fix applied:**
1. Set an explicit cluster-wide default write concern (`w: 1`) — required by MongoDB 5.0+ when a reconfig would change the implicit default write concern behavior.
2. Added the arbiter via `rs.addArb("mongo-arb-0.mongo-arb.mongodb.svc.cluster.local:27017")`.

**Result:**
- `votingMembersCount: 3` (was 2)
- `majorityVoteCount: 2` (now only 2 of 3 votes needed — elections succeed even if one data member goes down)
- Arbiter state: `ARBITER` (no storage used)
- App database `app.test` still has 3 documents — data is intact.
