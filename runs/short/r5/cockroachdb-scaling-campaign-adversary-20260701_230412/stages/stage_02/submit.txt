The CockroachDB cluster is fully operational. Here's a summary of what happened and the current state:

**Root cause (diagnosed):** The StatefulSet uses `podManagementPolicy: Parallel`, which launches all 3 pods simultaneously. The headless service DNS entries (`crdb-cluster-{0,1,2}.crdb-cluster.cockroachdb.svc.cluster.local`) aren't registered until each pod's readiness probe passes — so when pods first started, they couldn't resolve each other's hostnames, producing the `no such host` errors in the join/advertise logs. The nodes kept retrying; once DNS propagated (~3-4 min), all 3 connected. `cockroach init` had already been executed (evidently before this session), bootstrapping node-0, after which nodes 1 and 2 joined automatically.

**Current state — fully healthy:**
- All 3 nodes: `is_available: true`, `is_live: true`
- All 3 nodes visible in gossip (`node_id` 1, 2, 3)
- SQL is accepting connections and DDL writes work (created/dropped a test database)
- No `cockroach init` needed — it was already run and the cluster self-healed once DNS resolved
