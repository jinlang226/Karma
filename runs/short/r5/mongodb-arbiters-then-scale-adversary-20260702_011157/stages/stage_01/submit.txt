Done. Here's what was done and why:

**Problem:** The replica set had only 2 voting members (`mongo-rs-0` PRIMARY, `mongo-rs-1` SECONDARY). With 2 voters, majority = 2, meaning both members must be reachable for elections to succeed — losing either member makes the set unavailable.

**Fix:** Added `mongo-arb-0` as an arbiter to the replica set.

1. Set the cluster-wide default write concern to `{w: "majority"}` explicitly — required by MongoDB 5+ before any reconfig that changes the implicit default write concern.
2. Called `rs.addArb("mongo-arb-0.mongo-arb.mongodb.svc.cluster.local:27017")`.

**Result:**
- `votingMembersCount: 3`, `majorityVoteCount: 2`
- Elections now succeed with any 2 of the 3 voters — one data member can be lost and the set remains writable
- No storage cost added: the arbiter uses `emptyDir` and holds no data
- `app.test` collection: 3 documents intact, unchanged
