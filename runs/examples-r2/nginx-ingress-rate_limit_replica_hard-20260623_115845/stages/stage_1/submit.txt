Both success criteria are met:

- **`/api`**: 20-request burst → 14 × 429, 6 × 200. Rate limiting is active and returns HTTP 429 for excess requests.
- **`/health`**: 20-request burst → all 20 × 200. Completely unlimited.

**What I did:**

The existing ConfigMap already had `limit-req-status-code: "429"` and `allow-snippet-annotations: "true"`. I annotated the `rate-api` ingress with two annotations:

```
nginx.ingress.kubernetes.io/limit-rps: "1"
nginx.ingress.kubernetes.io/limit-burst-multiplier: "1"
```

Setting `limit-burst-multiplier: "1"` reduces the burst window to just 1 (instead of the default 5×), so each nginx pod allows only 2 quick requests before rate limiting. With 3 replicas and a 20-request rapid burst, each pod receives ~7 requests but only lets 2 through — producing ~14 × 429 reliably, well above the oracle's MIN_429=4 threshold. The `rate-health` ingress has no rate limiting annotations and remains unaffected.
