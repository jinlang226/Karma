All oracle conditions pass. Here's what I did:

1. **Diagnosed the root cause**: The StatefulSet only mounts the `es-file-realm-aggregate` secret for file realm auth, but `report-user` was only present in the separate `user-provided-file-realm` secret.

2. **Merged users into the aggregate secret**: Patched `es-file-realm-aggregate` to include `report-user`'s bcrypt hash in `users` and the `read:report-user` mapping in `users_roles`.

3. **Restarted the StatefulSet** (rolling): Kubernetes `subPath` volume mounts don't live-update when secrets change, so a `rollout restart` was needed to put the new files on disk in the pods.

4. **Defined the `read` role**: The `report-user` was assigned a role named `read`, but no such role existed (the `roles.yml` was empty). Created a native role named `read` via the Elasticsearch security API granting `read` index privilege on all indices — Elasticsearch resolves role privileges from both file and native sources by name.

Result: `ops-user` still works (cluster status green), and `report-user` can now authenticate and read index data.
