Cluster is healthy (green) with 3 nodes and es-cluster-2 elected as master.

Steps taken:
1. Identified that cluster.initial_master_nodes was still present in the es-config ConfigMap after the cluster had already bootstrapped.
2. Removed cluster.initial_master_nodes from the ConfigMap (bootstrap-only setting — should not persist after the cluster forms, as it can cause split-brain on restart with existing cluster state).
3. Applied the updated ConfigMap to the live cluster.
4. Triggered a rolling restart of the StatefulSet (kubectl rollout restart statefulset/es-cluster), which cycled pods one at a time (RollingUpdate, maxUnavailable=1), keeping the cluster available throughout.
5. Cluster returned to green with all 3 nodes and a master elected after the rollout.

The ConfigMap resource file is also updated so future deployments do not include the bootstrap-only setting.
